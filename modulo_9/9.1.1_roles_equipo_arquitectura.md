# Roles del Equipo de Arquitectura

**Tiempo estimado**: 30 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Estructura del Repositorio (8.2.2)

---

## Por que importa este subtema?

Una practica de arquitectura empresarial efectiva requiere roles claramente definidos. Sin una estructura organizacional adecuada, la arquitectura queda difusa entre diferentes equipos, perdiendo coherencia y efectividad.

## Roles Principales

### 1. Chief Architect / Enterprise Architect

```markdown
## Chief Architect

### Responsabilidades
| Area | Descripcion |
|------|-------------|
| Estrategia | Definir vision y estrategia de arquitectura |
| Governance | Establecer politicas y estandares |
| Alineacion | Asegurar alineacion IT-Negocio |
| Portafolio | Oversight del portafolio de arquitectura |
| Transformacion | Liderar iniciativas de transformacion |
| Stakeholders | Comunicar con C-suite y board |

### Competencias Clave
- Vision estrategica
- Comunicacion ejecutiva
- Liderazgo de cambio
- Pensamiento sistemico
- Influencia sin autoridad

### Metricas de Exito
| Metrica | Descripcion |
|---------|-------------|
| Alineacion | % de proyectos alineados con EA |
| Reutilizacion | Nivel de reutilizacion de building blocks |
| Time-to-market | Reduccion en tiempo de entrega |
| Deuda tecnica | Tendencia de deuda tecnica |
| Satisfaccion | NPS de stakeholders internos |

### Reporte
- Reporta a: CIO / CTO
- Supervisa: Domain Architects
- Colabora con: Business Leadership, PMO, Security
```

### 2. Domain Architects

#### Business Architect

```markdown
## Business Architect

### Responsabilidades
- Modelar capacidades de negocio
- Documentar procesos de negocio
- Identificar gaps de capacidad
- Traducir estrategia a arquitectura
- Definir business information model

### Entregables Tipicos
| Entregable | Proposito |
|------------|-----------|
| Capability Map | Vista de capacidades de negocio |
| Process Models | Documentacion de procesos |
| Value Streams | Flujos de valor del negocio |
| Organization Map | Estructura organizacional |
| Business Scenarios | Casos de uso de negocio |

### Competencias
- Modelado de negocio (ArchiMate, BPMN)
- Analisis de procesos
- Conocimiento del dominio
- Facilitacion de workshops
- Business analysis

### Colaboracion
- Trabaja con: Lideres de negocio, Product Owners
- Input de: Estrategia corporativa
- Output a: Application Architects
```

#### Data Architect

```markdown
## Data Architect

### Responsabilidades
- Disenar modelos de datos empresariales
- Definir data governance
- Establecer master data management
- Disenar data lakes/warehouses
- Asegurar calidad de datos

### Entregables Tipicos
| Entregable | Proposito |
|------------|-----------|
| Conceptual Data Model | Entidades de negocio |
| Logical Data Model | Estructura de datos |
| Data Flow Diagrams | Movimiento de datos |
| Data Dictionary | Definiciones de datos |
| MDM Design | Master data architecture |

### Competencias
- Modelado de datos (ERD, dimensional)
- Data governance
- Data quality management
- SQL y bases de datos
- Big data technologies

### Colaboracion
- Trabaja con: Data Engineers, DBAs
- Input de: Business Architects
- Output a: Application Architects, BI Team
```

#### Application Architect

```markdown
## Application Architect

### Responsabilidades
- Disenar portafolio de aplicaciones
- Definir patrones de integracion
- Establecer estandares de desarrollo
- Evaluar y seleccionar tecnologias
- Guiar modernizacion de aplicaciones

### Entregables Tipicos
| Entregable | Proposito |
|------------|-----------|
| Application Portfolio | Inventario de aplicaciones |
| Integration Architecture | Patrones de integracion |
| API Design Guidelines | Estandares de APIs |
| Reference Architectures | Arquitecturas de referencia |
| Technology Radar | Evaluacion de tecnologias |

### Competencias
- Patrones de arquitectura (microservices, SOA)
- APIs y integracion
- Cloud platforms
- Development practices
- Application modernization

### Colaboracion
- Trabaja con: Development Teams, DevOps
- Input de: Business/Data Architects
- Output a: Technology Architects, Dev Teams
```

#### Technology Architect

```markdown
## Technology Architect

### Responsabilidades
- Disenar infraestructura de TI
- Definir plataformas y estandares
- Establecer arquitectura cloud
- Disenar redes y seguridad infra
- Planificar capacidad y escalabilidad

### Entregables Tipicos
| Entregable | Proposito |
|------------|-----------|
| Infrastructure Design | Diseno de infraestructura |
| Cloud Architecture | Arquitectura cloud |
| Network Design | Diseno de red |
| Platform Standards | Estandares de plataforma |
| Capacity Plan | Plan de capacidad |

### Competencias
- Cloud platforms (AWS, Azure, GCP)
- Infrastructure as Code
- Networking
- Containers y Kubernetes
- Performance engineering

### Colaboracion
- Trabaja con: DevOps, SRE, Network Team
- Input de: Application Architects
- Output a: Operations, Security
```

#### Security Architect

```markdown
## Security Architect

### Responsabilidades
- Definir estrategia de seguridad
- Disenar controles de seguridad
- Establecer estandares de seguridad
- Evaluar riesgos de arquitectura
- Asegurar compliance

### Entregables Tipicos
| Entregable | Proposito |
|------------|-----------|
| Security Reference Architecture | Patrones de seguridad |
| Threat Models | Analisis de amenazas |
| Security Standards | Estandares y controles |
| Risk Assessments | Evaluacion de riesgos |
| Compliance Mapping | Mapeo regulatorio |

### Competencias
- Security frameworks (NIST, ISO27001)
- Threat modeling
- Identity & Access Management
- Cryptography
- Compliance y regulaciones

### Colaboracion
- Trabaja con: Security Operations, Compliance
- Input de: Todos los domain architects
- Output a: Development Teams, Operations
```

### 3. Solution Architect

```markdown
## Solution Architect

### Responsabilidades
- Disenar soluciones para proyectos especificos
- Traducir requisitos a arquitectura tecnica
- Asegurar adherencia a estandares EA
- Guiar equipos de desarrollo
- Resolver problemas tecnicos complejos

### Entregables Tipicos
| Entregable | Proposito |
|------------|-----------|
| Solution Architecture Document | Diseno de solucion |
| Technical Design | Diseno tecnico detallado |
| Integration Design | Diseno de integraciones |
| Deployment Architecture | Arquitectura de deployment |
| Technical Decisions (ADRs) | Decisiones documentadas |

### Competencias
- Multiples tecnologias (full-stack)
- Diseno de sistemas
- Problem solving
- Communication
- Hands-on cuando necesario

### Colaboracion
- Trabaja con: Development Teams, QA
- Input de: Domain Architects, Product Owners
- Output a: Development Teams
- Reporta a: Chief Architect o Project Manager

### Tipos de Solution Architects
| Tipo | Enfoque |
|------|---------|
| Generalist | Multiples dominios |
| Domain Expert | Especializado (SAP, Salesforce, etc) |
| Integration Specialist | Integraciones complejas |
| Cloud Specialist | Soluciones cloud-native |
```

## Architecture Board

```markdown
## Architecture Board

### Composicion
| Miembro | Rol en Board |
|---------|--------------|
| Chief Architect | Chair |
| Business Architect | Representante negocio |
| Data Architect | Representante datos |
| Application Architect | Representante apps |
| Technology Architect | Representante infra |
| Security Architect | Representante seguridad |
| Enterprise PMO | Representante proyectos |

### Responsabilidades
- Aprobar cambios arquitectonicos mayores
- Revisar compliance de proyectos
- Resolver conflictos de arquitectura
- Aprobar excepciones y waivers
- Evolucionar estandares y principios

### Operacion
| Aspecto | Detalle |
|---------|---------|
| Frecuencia | Quincenal |
| Quorum | 4 de 6 domain architects |
| Decisiones | Consenso, mayoria si necesario |
| Documentacion | Actas publicadas |
```

## Modelos de Organizacion

### Modelo Centralizado

```
+-------------------+
| Enterprise        |
| Architecture      |
| Team              |
| (10-20 architects)|
+-------------------+
        |
        v
  All projects
```

**Pros**: Consistencia, economias de escala
**Contras**: Puede ser cuello de botella

### Modelo Federado

```
        +-----------+
        | EA Team   |
        | (small)   |
        +-----+-----+
              |
    +---------+---------+
    |         |         |
    v         v         v
+-------+ +-------+ +-------+
| BU 1  | | BU 2  | | BU 3  |
| Arch  | | Arch  | | Arch  |
+-------+ +-------+ +-------+
```

**Pros**: Cercania al negocio, agilidad
**Contras**: Riesgo de inconsistencia

### Modelo Hibrido

```
        +-------------+
        | Central EA  |
        | (Governance)|
        +------+------+
               |
    +----------+----------+
    |          |          |
    v          v          v
+--------+ +--------+ +--------+
| Domain | | Domain | | Domain |
| Team 1 | | Team 2 | | Team 3 |
+--------+ +--------+ +--------+
    |          |          |
    v          v          v
Solution Architects embedded in projects
```

**Pros**: Balance entre consistencia y agilidad
**Contras**: Requiere coordinacion efectiva

## Resumen

| Rol | Enfoque | Nivel |
|-----|---------|-------|
| Chief Architect | Estrategia y governance | Enterprise |
| Business Architect | Capacidades y procesos | Domain |
| Data Architect | Modelos y governance de datos | Domain |
| Application Architect | Apps e integracion | Domain |
| Technology Architect | Infraestructura y cloud | Domain |
| Security Architect | Seguridad y compliance | Cross-cutting |
| Solution Architect | Proyectos especificos | Solution |
