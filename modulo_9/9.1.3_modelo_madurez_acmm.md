# Modelo de Madurez de Arquitectura (ACMM)

**Tiempo estimado**: 30 minutos
**Nivel**: Intermedio-Avanzado
**Prerrequisitos**: Skills y Desarrollo (9.1.2)

---

## Por que importa este subtema?

El Architecture Capability Maturity Model (ACMM) permite evaluar que tan madura es la practica de arquitectura empresarial en una organizacion. Conocer el nivel actual y el objetivo guia las inversiones en mejorar la capacidad arquitectonica.

---

## Que es el ACMM

> "Un framework para evaluar y mejorar la capacidad de una organizacion para desarrollar y usar arquitectura empresarial de manera efectiva."

**Propositos**:
- Evaluar estado actual de la practica EA
- Identificar areas de mejora
- Planificar evolucion de la capacidad
- Benchmark contra mejores practicas
- Comunicar progreso a stakeholders

---

## Niveles de Madurez

### Vista General

```
Nivel 5: OPTIMIZING
         Mejora continua, innovacion, industry leader
              ^
              |
Nivel 4: MANAGED
         Metricas, gobierno efectivo, valor demostrado
              ^
              |
Nivel 3: DEFINED
         Procesos estandarizados, roles claros, repositorio
              ^
              |
Nivel 2: DEVELOPING
         Procesos basicos, equipo inicial, herramientas
              ^
              |
Nivel 1: INITIAL
         Ad-hoc, dependiente de individuos, reactivo
              ^
              |
Nivel 0: NONE
         Sin practica de arquitectura
```

## Descripcion de Niveles

### Nivel 0: None

```markdown
## Nivel 0: Sin Arquitectura

### Caracteristicas
- No existe funcion de arquitectura
- No hay artefactos de arquitectura
- Decisiones tecnologicas descentralizadas
- Cada proyecto decide independientemente
- No hay vision de portafolio

### Sintomas
- Duplicacion de sistemas
- Integraciones punto a punto
- Deuda tecnica alta
- Costos de TI crecientes
- Time-to-market lento
```

### Nivel 1: Initial

```markdown
## Nivel 1: Inicial

### Caracteristicas
- Arquitectura existe pero es ad-hoc
- Depende de individuos heroicos
- Procesos no documentados
- Herramientas inconsistentes
- Reactivo a problemas

### Indicadores
| Area | Estado |
|------|--------|
| Procesos | No definidos |
| Roles | Informales |
| Herramientas | Variables |
| Repositorio | No existe |
| Governance | Ninguno |
| Metricas | Ninguna |

### Typical Actions
- "Necesitamos un arquitecto para este proyecto"
- "Cada equipo usa sus propias herramientas"
- "No se donde encontrar la documentacion"
```

```markdown
## Nivel 2: En Desarrollo

### Caracteristicas
```

```markdown
- Roles y responsabilidades claros

### Indicadores
| Area | Estado |
|------|--------|

| Roles | Completos, capacitados |
| Herramientas | Adoptadas, integradas |

- ADM ejecutado regularmente

```

### Nivel 4: Managed

```markdown
## Nivel 4: Gestionado

### Caracteristicas
- Metricas de arquitectura definidas
- Decisiones basadas en datos
- Valor de EA demostrado
- Mejora basada en feedback
- Alineacion con estrategia evidente

### Indicadores
| Area | Estado |
|------|--------|
| Procesos | Medidos, optimizados |
| Roles | Maduros, especializados |
| Herramientas | Optimizadas |
| Repositorio | Completo, actualizado |

| Metrica | Ejemplo |
|---------|---------|
| Time-to-market | -30% vs baseline |
```

## Proceso de Assessment

### Pasos

```
1. PLANIFICAR
   - Definir alcance
   - Seleccionar assessors
   - Programar actividades
        |
        v
2. RECOPILAR DATOS
   - Entrevistas
   - Revision de documentos
   - Observacion de practicas
        |
        v
3. EVALUAR
   - Scoring por dimension
   - Identificar gaps
   - Validar hallazgos
        |
        v
4. REPORTAR
   - Nivel actual por dimension
   - Fortalezas y debilidades
   - Recomendaciones
        |
        v
5. PLANIFICAR MEJORA
   - Definir nivel objetivo
   - Roadmap de mejora
   - Quick wins y largo plazo
```

```markdown
## Architecture Maturity Assessment Report

### Resumen Ejecutivo
| Dimension | Nivel Actual | Objetivo | Gap |

| Organizacion | 2 | 3 | +1 |
| Informacion | 1 | 3 | +2 |
| Gobierno | 2 | 3 | +1 |
| Tecnologia | 2 | 3 | +1 |
| Cultura | 2 | 3 | +1 |
| **Promedio** | **1.8** | **3.0** | **+1.2** |

### Nivel Global: 2 (En Desarrollo)

### Hallazgos por Dimension

#### Procesos (Nivel 2)
**Fortalezas**:
- ADM adoptado como framework
- Proceso de design review en lugar

**Debilidades**:
- No hay metricas de proceso
- Proceso no seguido consistentemente

**Recomendaciones**:
- Documentar proceso formalmente
- Establecer checkpoints obligatorios
- Definir KPIs de proceso

#### Informacion (Nivel 1)
**Fortalezas**:
- Algunos artefactos existen

**Debilidades**:
- Repositorio fragmentado
- Documentacion desactualizada
- Dificil encontrar artefactos

**Recomendaciones**:
- Implementar herramienta EA
- Definir estructura de repositorio
- Asignar ownership de contenido

### Roadmap de Mejora

| Fase | Timeline | Objetivo | Acciones Clave |
|------|----------|----------|----------------|
| 1 | Q1-Q2 | Nivel 2.5 | Quick wins, herramienta |
| 2 | Q3-Q4 | Nivel 3.0 | Procesos, governance |
| 3 | Year 2 | Nivel 3.5 | Metricas, cultura |
| 4 | Year 3 | Nivel 4.0 | Optimizacion |
```
