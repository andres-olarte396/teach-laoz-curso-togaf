# Adaptacion del ADM

**Tiempo estimado**: 25 minutos
**Nivel**: Intermedio-Avanzado
**Prerrequisitos**: Inputs, Outputs y Steps (2.1.2)

---

## Por que importa este subtema?

TOGAF no es un proceso rigido: esta disenado para ser adaptado. Una startup de 50 personas no puede ejecutar el mismo ADM que un banco con 50,000 empleados. El exito de TOGAF depende de saber **que adaptar** y **como adaptarlo** sin perder el valor del framework.

---

## Principios de Adaptacion

### Que se puede adaptar

| Elemento | Adaptable? | Ejemplo |
|----------|-----------|---------|
| Fases | Si, pueden omitirse o combinarse | Combinar B+C+D en una sola iteracion |
| Steps | Si, pueden simplificarse | Reducir 11 steps de Fase A a 5 |
| Outputs | Si, pueden simplificarse | Documento de 50 paginas -> 10 paginas |
| Terminologia | Si, puede ajustarse | "ABB" -> "Componente de Arquitectura" |
| Herramientas | Si, son opcionales | Usar Confluence en lugar de herramienta EA |

### Que NO se debe adaptar

| Elemento | Por que |
|----------|---------|
| Secuencia logica | A debe preceder B-D; E necesita gaps de B-D |
| Gestion de requisitos | Es el pegamento del proceso |
| Governance minimo | Sin governance, no hay conformidad |
| Documentacion de decisiones | Sin registro, no hay trazabilidad |

---

## Factores que Determinan la Adaptacion

### 1. Tamano de la Organizacion

| Tamano | Adaptacion |
|--------|------------|
| **Pequena** (<100 personas) | ADM "ligero", 1 arquitecto, ciclos de semanas |
| **Mediana** (100-1000) | ADM estandar, equipo pequeno, ciclos de meses |
| **Grande** (>1000) | ADM completo, multiples equipos, ciclos de anos |

### 2. Madurez de Arquitectura

| Madurez | Adaptacion |
|---------|------------|
| **Inicial** | Enfocarse en Preliminar + A, establecer bases |
| **En desarrollo** | ADM completo con simplificaciones |
| **Establecida** | ADM completo, enfasis en Governance (G, H) |

### 3. Alcance del Proyecto

| Alcance | Adaptacion |
|---------|------------|
| **Enterprise** | ADM completo, todas las fases |
| **Segment** | Omitir Preliminar si ya existe, fases mas cortas |
| **Capability** | Solo A + fases relevantes + E/F |

### 4. Restriccion de Tiempo

| Tiempo | Adaptacion |
|--------|------------|
| **Anos** | ADM completo con iteraciones |
| **Meses** | Fases condensadas, outputs simplificados |
| **Semanas** | "Architecture Sprint" - minimo viable |

## Patrones de Adaptacion Comunes

### Patron 1: ADM para Startup/Agile

```
CONTEXTO: Empresa de 80 personas, cultura agil, necesita
          arquitectura para escalar de 1M a 10M usuarios.

ADAPTACION:

Fase Preliminar: OMITIR
  - No hay framework existente que adaptar
  - Principios emergen organicamente

Fase A: 1 semana
  - Workshop de 2 dias con founders + tech leads
  - Vision en 1 pagina
  - Statement of Work verbal

Fases B+C+D: COMBINADAS, 2-3 semanas
  - Un solo documento "Architecture Overview"
  - Diagramas C4 en lugar de artefactos TOGAF completos
  - Gap analysis informal

Fase E+F: COMBINADAS, 1 semana
  - Roadmap en Miro/Notion
  - Sin Transition Architectures formales

Fase G: CONTINUA
  - Architectural Decision Records (ADRs)
  - Revisiones en pull requests

Fase H: INTEGRADA
  - Parte del proceso de refinamiento continuo
```

### Patron 2: ADM para Iniciativa Estrategica (Enterprise)

```
CONTEXTO: Banco con 30,000 empleados, iniciativa de
          transformacion digital de 3 anos.

ADAPTACION:

Fase Preliminar: 3 meses
  - Establecer Architecture Board
  - Contratar/capacitar equipo de 15 arquitectos
  - Implementar herramienta EA (Sparx, LeanIX, etc.)
  - Definir 25 principios de arquitectura

Fase A: 2 meses
  - Sponsor: CEO
  - Vision validada con 50+ stakeholders
  - Statement of Work aprobado por comite ejecutivo
  - Presupuesto de $50M asignado

Fases B, C, D: 4 meses cada una
  - Equipos dedicados por dominio
  - 100+ artefactos por dominio
  - Reviews formales con sign-off

Fase E: 2 meses
  - 200+ work packages identificados
  - Evaluacion de proveedores
  - RFPs emitidos

Fase F: 2 meses
  - Roadmap de 3 anos
  - 4 Transition Architectures
  - Plan de migracion de 500 paginas

Fases G+H: 30 meses
  - Governance continuo
  - Reviews trimestrales
  - Actualizaciones anuales de arquitectura
```

### Patron 3: ADM para Proyecto de Capacidad

```
CONTEXTO: Empresa mediana necesita implementar sistema
          de gestion de identidades (IAM).

ADAPTACION:

Fase Preliminar: OMITIR
  - Framework enterprise ya existe

Fase A: 3 dias
  - Alcance: Solo IAM
  - Vision: "Identidad unificada para 5000 usuarios"
  - Stakeholders: IT Security, HR, Legal

Fase B: REDUCIDA, 1 semana
  - Solo procesos de onboarding/offboarding
  - Solo roles relacionados con acceso

Fase C: 2 semanas
  - Arquitectura de datos de identidad
  - Integraciones con apps criticas (10 apps)

Fase D: 1 semana
  - Seleccion de plataforma (Okta vs Azure AD vs custom)
  - Infraestructura de soporte

Fases E+F: 1 semana
  - Roadmap de 6 meses
  - 3 fases: Core IAM -> SSO -> MFA

Fase G: 6 meses
  - Reviews mensuales
  - Ajustes segun feedback

Fase H: INTEGRADA con operaciones
```

## Tabla de Adaptacion Rapida

| Factor | Opcion Ligera | Opcion Estandar | Opcion Completa |
|--------|---------------|-----------------|-----------------|
| **Ciclo total** | 4-8 semanas | 4-8 meses | 12-24 meses |
| **Equipo** | 1-2 arquitectos | 3-8 arquitectos | 10+ arquitectos |
| **Documentacion** | 10-30 paginas | 100-300 paginas | 500+ paginas |
| **Fases** | A + (BCD combinadas) + (EF combinadas) | Todas, simplificadas | Todas, completas |
| **Stakeholders** | 5-10 | 20-50 | 50+ |
| **Governance** | Reviews informales | Board trimestral | Board mensual |
| **Herramientas** | Draw.io, Notion | Confluence, Archi | Enterprise tools |

---

## Errores Comunes en Adaptacion

| Error | Consecuencia | Solucion |
|-------|--------------|----------|
| Omitir Fase A | Proyectos sin direccion ni sponsor | A siempre es necesaria (aunque sea 1 dia) |
| Omitir gap analysis | Roadmap irreal | Gap analysis siempre, aunque sea informal |
| Eliminar governance | Implementaciones no conformes | Minimo: revisiones periodicas |
| Adaptar terminologia sin mapeo | Confusion | Crear glosario de equivalencias |
| No documentar adaptaciones | Inconsistencia | Documentar en Tailored Framework |

---

## Documentando la Adaptacion

Toda adaptacion debe documentarse en el **Tailored Architecture Framework**:

```markdown
# Tailored Architecture Framework - [Organizacion]

## Alcance de Aplicacion
- Proyectos estrategicos: ADM Completo
- Proyectos de segmento: ADM Simplificado
- Proyectos de capacidad: ADM Ligero

## Adaptaciones por Tipo

### ADM Simplificado
- Fases B, C, D pueden ejecutarse en paralelo
- Outputs reducidos a: Architecture Overview + Gap Analysis
- Governance: Reviews mensuales

### ADM Ligero
- Fases B+C+D combinadas
- Fases E+F combinadas
- Outputs: Architecture Decision Records + Roadmap
- Governance: Review al final de cada fase

## Terminologia Adaptada
| TOGAF | Nuestra organizacion |
|-------|---------------------|
| ABB | Componente de Arquitectura |
| Statement of Work | Acta de Inicio |
| Architecture Board | Comite de Arquitectura |

## Herramientas
- Modelado: Archi + Draw.io
- Documentacion: Confluence
- Governance: Jira

## Roles
| Rol TOGAF | Nuestro rol |
|-----------|-------------|
| Enterprise Architect | Director de Arquitectura |
| Domain Architect | Arquitecto de Solucion |
| Architecture Board | Comite Tecnico |
```

## Resumen

| Pregunta | Respuesta |
|----------|-----------|
| Se puede adaptar el ADM? | Si, es parte del diseno |
| Que se puede adaptar? | Fases, steps, outputs, terminologia |
| Que no se debe adaptar? | Secuencia logica, RM, governance minimo |
| Como documento la adaptacion? | En el Tailored Architecture Framework |
