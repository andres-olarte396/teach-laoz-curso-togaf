# Estructura del Ciclo ADM

**Tiempo estimado**: 30 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Modulo 1 completo (Fundamentos TOGAF)

---

## Por que importa este subtema?

El ADM (Architecture Development Method) es el proceso que distingue a TOGAF de otros frameworks. Mientras Zachman te da una taxonomia y ArchiMate te da una notacion, TOGAF te da un **proceso paso a paso** para crear arquitectura empresarial.

Sin entender la estructura del ADM, usaras TOGAF como una coleccion de plantillas desconectadas en lugar de un metodo coherente.

---

## Vision General del Ciclo

El ADM es un ciclo de 8 fases mas una fase preliminar y un proceso central de gestion de requisitos:

## Las Fases en Detalle

### Fase Preliminar: Framework y Principios

**Proposito**: Preparar la organizacion para ejecutar proyectos de arquitectura exitosos.

**Se ejecuta**: Una vez (o se actualiza periodicamente), no en cada proyecto.

| Actividad | Resultado |
|-----------|-----------|
| Definir alcance de la empresa | Scope of the Enterprise |
| Confirmar governance framework | Architecture Governance Framework |
| Definir equipo de arquitectura | Organization Model |
| Establecer principios | Architecture Principles Catalog |
| Adaptar TOGAF | Tailored Architecture Framework |

---

### Fase A: Vision de Arquitectura

**Proposito**: Establecer el proyecto de arquitectura, definir alcance, y obtener aprobacion.

**Salidas clave**:
- Architecture Vision
- Statement of Architecture Work (contrato del proyecto)
- Stakeholder Map

**Pregunta que responde**: "Que problema estamos resolviendo y por que importa?"

---

### Fases B, C, D: Arquitecturas de Dominio

Estas tres fases desarrollan las arquitecturas baseline y target en cada dominio:

| Fase | Dominio | Pregunta que responde |
|------|---------|----------------------|
| **B** | Negocio | Como opera el negocio hoy? Como deberia operar? |
| **C** | Datos y Aplicaciones | Que datos necesitamos? Que aplicaciones los gestionan? |
| **D** | Tecnologia | Que infraestructura soporta las aplicaciones y datos? |

**Importante**: Pueden ejecutarse en paralelo o secuencialmente segun el contexto.

```
Opcion 1: Secuencial        Opcion 2: Paralelo
(Negocio primero)           (Equipos separados)

    B                           B
    |                          /|\
    v                         / | \
    C                        /  |  \
    |                       C   C   D
    v                        \  |  /
    D                         \ | /
                               E
```

---

### Fase E: Oportunidades y Soluciones

**Proposito**: Planificacion inicial de la implementacion. Identificar proyectos y work packages.

**Actividades**:
- Consolidar gaps de todas las arquitecturas de dominio
- Identificar Work Packages
- Evaluar opciones (build vs buy vs reuse)
- Agrupar trabajo en proyectos

**Salida clave**: Implementation and Migration Strategy

---

### Fase F: Planificacion de Migracion

**Proposito**: Crear el roadmap detallado de implementacion.

**Actividades**:
- Priorizar proyectos
- Definir Transition Architectures
- Crear Architecture Roadmap
- Desarrollar Implementation and Migration Plan

**Salida clave**: Architecture Roadmap con fechas y dependencias

---

### Fase G: Gobierno de Implementacion

**Proposito**: Asegurar que la implementacion sea conforme a la arquitectura.

**Actividades**:
- Establecer Architecture Contracts con equipos de implementacion
- Ejecutar Compliance Reviews
- Gestionar excepciones y dispensas

**Salida clave**: Architecture Contracts, Compliance Assessments

---

### Fase H: Gestion de Cambios de Arquitectura

**Proposito**: Gestionar cambios a la arquitectura de manera controlada.

**Actividades**:
- Monitorear cambios en el entorno (tecnologia, negocio, regulacion)
- Evaluar impacto de cambios
- Decidir si iniciar nuevo ciclo ADM

**Salida clave**: Change Requests, Architecture Updates

---

### Gestion de Requisitos: El Proceso Central

**No es una fase secuencial** sino un proceso que opera continuamente:

```
+----------------------------------------------------------+
|               GESTION DE REQUISITOS                       |
|                                                           |
|  +--------+    +--------+    +--------+    +--------+    |
|  | Fase A | -> | Fase B | -> | Fase C | -> | ...    |    |
|  +---+----+    +---+----+    +---+----+    +--------+    |
|      |             |             |                        |
|      v             v             v                        |
|  +-----------------------------------------------+       |
|  |          REQUIREMENTS REPOSITORY              |       |
|  |  - Requisitos de negocio                      |       |
|  |  - Requisitos de datos                        |       |
|  |  - Requisitos de aplicaciones                 |       |
|  |  - Requisitos tecnologicos                    |       |
|  |  - Restricciones                              |       |
|  +-----------------------------------------------+       |
+----------------------------------------------------------+
```

**Funciones**:
- Capturar requisitos en cualquier fase
- Mantener trazabilidad requisito <-> arquitectura
- Gestionar cambios en requisitos
- Priorizar requisitos

## El Ciclo es Iterativo

### Iteracion dentro de una fase

```
Fase B: Arquitectura de Negocio
+-----------------------------------+
|                                   |
|   Draft 1 --> Review --> Draft 2  |
|       ^                    |      |
|       |                    |      |
|       +--------------------+      |
|                                   |
+-----------------------------------+
```

### Iteracion entre fases

```
Fase B          Fase C          Fase D
   |               |               |
   +----> E <------+               |
          |                        |
          | Descubrimiento:        |
          | "Necesitamos datos     |
          |  que no modelamos"     |
          |                        |
          +-----> Regresar a C ----+
```

### Iteracion del ciclo completo

```
Ciclo 1: Enterprise Architecture (18 meses)
    Prelim -> A -> B -> C -> D -> E -> F
                                      |
Ciclo 2: Segment (Marketing) (6 meses)|
         A -> B -> C -> D -> E -> F --+
                                      |
Ciclo 3: Capability (CRM) (8 semanas) |
              A -> C -> E -> F -------+
```

## Resumen de Fases

| Fase | Nombre | Duracion Tipica | Output Principal |
|------|--------|-----------------|------------------|
| Prelim | Framework | 1-3 meses (una vez) | Tailored Framework |
| A | Vision | 2-4 semanas | Architecture Vision |
| B | Negocio | 4-8 semanas | Business Architecture |
| C | SI | 4-8 semanas | Data & App Architecture |
| D | Tecnologia | 4-8 semanas | Technology Architecture |
| E | Oportunidades | 2-4 semanas | Migration Strategy |
| F | Migracion | 2-4 semanas | Roadmap |
| G | Gobierno | Durante implementacion | Compliance Assessments |
| H | Cambios | Continuo | Change Requests |
