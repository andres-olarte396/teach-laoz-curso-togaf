# Inputs, Outputs y Steps del ADM

**Tiempo estimado**: 30 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Estructura del Ciclo ADM (2.1.1)

---

## Por que importa este subtema?

Cada fase del ADM tiene una estructura consistente: recibe inputs, ejecuta steps, y produce outputs. Conocer esta estructura te permite:

1. **Planificar**: Saber que necesitas antes de iniciar una fase
2. **Ejecutar**: Seguir los pasos definidos
3. **Verificar**: Confirmar que produjiste los outputs requeridos
4. **Comunicar**: Explicar a stakeholders que esperar

---

## Estructura General de una Fase

```
+------------------------------------------+
|              FASE [X]                     |
+------------------------------------------+
|                                          |
|  INPUTS                                  |
|  - Outputs de fases anteriores           |
|  - Materiales de referencia              |
|  - Requisitos actualizados               |
|                                          |
|  STEPS                                   |
|  1. Paso 1                               |
|  2. Paso 2                               |
|  ...                                     |
|  n. Paso n                               |
|                                          |
|  OUTPUTS                                 |
|  - Entregables de la fase                |
|  - Actualizaciones al repositorio        |
|  - Requisitos nuevos/refinados           |
|                                          |
+------------------------------------------+
```

## Fase Preliminar

### Inputs

| Input | Descripcion |
|-------|-------------|
| TOGAF Library | Estandar TOGAF y materiales de referencia |
| Other frameworks | Frameworks existentes en la organizacion |
| Board strategies | Estrategia de negocio y direccion ejecutiva |
| IT strategies | Estrategia de TI existente |
| Business principles | Principios de negocio de la organizacion |

### Steps

1. Definir la organizacion de arquitectura
2. Identificar y establecer principios de arquitectura
3. Adaptar TOGAF y otros frameworks
4. Implementar herramientas de arquitectura

### Outputs

| Output | Descripcion |
|--------|-------------|
| Organizational Model for EA | Estructura del equipo de arquitectura |
| Tailored Architecture Framework | TOGAF adaptado a la organizacion |
| Architecture Principles | Catalogo de principios |
| Architecture Repository | Repositorio inicial configurado |
| Business/IT strategies | Estrategias refinadas |

---

## Fase A: Vision de Arquitectura

### Inputs

| Input | Descripcion |
|-------|-------------|
| Request for Architecture Work | Solicitud que inicia el proyecto |
| Business principles, goals, drivers | Contexto de negocio |
| Architecture Repository | Contenido existente del repositorio |
| Tailored Architecture Framework | Framework adaptado (de Preliminar) |
| Architecture Principles | Principios establecidos |

### Steps

1. Establecer el proyecto de arquitectura
2. Identificar stakeholders, concerns, y requisitos de negocio
3. Confirmar y elaborar business goals, drivers, y restricciones
4. Evaluar capabilities de negocio
5. Evaluar readiness para transformacion de negocio
6. Definir alcance
7. Confirmar y elaborar principios de arquitectura
8. Desarrollar Architecture Vision
9. Definir proposiciones de valor de la Target Architecture
10. Identificar riesgos de transformacion y mitigaciones
11. Desarrollar Statement of Architecture Work; asegurar aprobacion

### Outputs

| Output | Descripcion |
|--------|-------------|
| Approved Statement of Architecture Work | Contrato del proyecto aprobado |
| Architecture Vision | Documento de vision |
| Refined business principles, goals, drivers | Contexto refinado |
| Architecture Principles (refined) | Principios actualizados |
| Stakeholder Map | Mapa de interesados |
| Communications Plan | Plan de comunicaciones |
| Requirements Repository (populated) | Requisitos iniciales |

---

## Fase B: Arquitectura de Negocio

### Inputs

| Input | Descripcion |
|-------|-------------|
| Architecture Vision | De Fase A |
| Statement of Architecture Work | De Fase A |
| Architecture Repository | Contenido actualizado |
| Draft Architecture Definition Document | Borrador inicial |
| Architecture Requirements Specification | Requisitos capturados |

### Steps

1. Seleccionar modelos de referencia, viewpoints, y herramientas
2. Desarrollar descripcion de Baseline Business Architecture
3. Desarrollar descripcion de Target Business Architecture
4. Realizar Gap Analysis
5. Definir componentes candidatos del roadmap
6. Resolver impactos en el landscape de arquitectura
7. Conducir revision formal con stakeholders
8. Finalizar Business Architecture
9. Crear Architecture Definition Document (porcion de negocio)

### Outputs

| Output | Descripcion |
|--------|-------------|
| Draft Architecture Definition Document (Business) | Arquitectura de negocio documentada |
| Draft Architecture Requirements Specification | Requisitos refinados |
| Business Architecture components of Roadmap | Componentes para el roadmap |

---

## Fase C: Arquitectura de Sistemas de Informacion

### Inputs

Similares a Fase B, mas la arquitectura de negocio completada.

### Steps (para Datos)

1. Seleccionar modelos de referencia, viewpoints, y herramientas
2. Desarrollar descripcion de Baseline Data Architecture
3. Desarrollar descripcion de Target Data Architecture
4. Realizar Gap Analysis
5. Definir componentes candidatos del roadmap
6. Resolver impactos en el landscape de arquitectura
7. Conducir revision formal con stakeholders
8. Finalizar Data Architecture
9. Crear Architecture Definition Document (porcion de datos)

### Steps (para Aplicaciones)

Pasos similares aplicados al dominio de aplicaciones.

### Outputs

| Output | Descripcion |
|--------|-------------|
| Draft Architecture Definition Document (Data) | Arquitectura de datos |
| Draft Architecture Definition Document (Application) | Arquitectura de aplicaciones |
| Data Architecture components of Roadmap | Componentes de datos para roadmap |
| Application Architecture components of Roadmap | Componentes de apps para roadmap |

---

## Fase D: Arquitectura Tecnologica

### Inputs

Outputs de fases A, B, C mas el contenido del repositorio.

### Steps

1. Seleccionar modelos de referencia, viewpoints, y herramientas
2. Desarrollar descripcion de Baseline Technology Architecture
3. Desarrollar descripcion de Target Technology Architecture
4. Realizar Gap Analysis
5. Definir componentes candidatos del roadmap
6. Resolver impactos en el landscape de arquitectura
7. Conducir revision formal con stakeholders
8. Finalizar Technology Architecture
9. Crear Architecture Definition Document (porcion de tecnologia)

### Outputs

| Output | Descripcion |
|--------|-------------|
| Draft Architecture Definition Document (Technology) | Arquitectura tecnologica |
| Technology Architecture components of Roadmap | Componentes de tecnologia para roadmap |

---

## Fase E: Oportunidades y Soluciones

### Inputs

| Input | Descripcion |
|-------|-------------|
| Architecture Vision | De Fase A |
| Draft Architecture Definition Document | De Fases B, C, D |
| Draft Architecture Requirements Specification | Requisitos consolidados |
| Capability Assessment | Evaluacion de capacidades |

### Steps

1. Determinar/confirmar atributos clave de cambio corporativo
2. Determinar restricciones de negocio para implementacion
3. Revisar y consolidar Gap Analysis de Fases B a D
4. Revisar requisitos consolidados en relacion a gaps
5. Consolidar y reconciliar requisitos de interoperabilidad
6. Refinar y validar dependencias
7. Confirmar readiness y riesgo para transformacion de negocio
8. Formular Implementation and Migration Strategy
9. Identificar y agrupar work packages
10. Identificar Transition Architectures
11. Crear Architecture Roadmap y Implementation and Migration Plan

### Outputs

| Output | Descripcion |
|--------|-------------|
| Implementation and Migration Strategy | Estrategia de implementacion |
| Architecture Roadmap (initial) | Roadmap inicial |
| Transition Architectures (if any) | Arquitecturas de transicion |
| Implementation Factor Assessment | Evaluacion de factores |

---

## Fase F: Planificacion de Migracion

### Inputs

Outputs de Fase E mas informacion de implementacion.

### Steps

1. Confirmar interacciones para Implementation and Migration Plan con frameworks de gestion
2. Asignar valor de negocio a cada work package
3. Estimar requerimientos de recursos, tiempos, y costos
4. Priorizar proyectos de migracion
5. Confirmar Architecture Roadmap y actualizar Architecture Definition Document
6. Completar el ciclo de desarrollo de arquitectura
7. Generar Implementation and Migration Plan
8. Document lessons learned

### Outputs

| Output | Descripcion |
|--------|-------------|
| Implementation and Migration Plan | Plan detallado |
| Architecture Definition Document (finalized) | Documento final |
| Architecture Requirements Specification (finalized) | Requisitos finales |
| Architecture Roadmap (finalized) | Roadmap final |
| Transition Architectures (finalized) | Transiciones finales |
| Reusable Architecture Building Blocks | BBs reutilizables |

---

## Fase G: Gobierno de Implementacion

### Inputs

| Input | Descripcion |
|-------|-------------|
| Implementation and Migration Plan | De Fase F |
| Architecture Definition Document | Documento final |
| Architecture Requirements Specification | Requisitos finales |
| Architecture Contracts | Contratos de fase anterior |

### Steps

1. Confirmar alcance y prioridades para implementacion
2. Identificar recursos y skills de implementacion
3. Guiar desarrollo de Solutions Architecture
4. Ejecutar compliance reviews
5. Implementar operaciones de negocio y TI
6. Ejecutar post-implementation review y cerrar implementacion

### Outputs

| Output | Descripcion |
|--------|-------------|
| Architecture Contracts | Contratos firmados |
| Compliance Assessments | Evaluaciones de conformidad |
| Change Requests | Solicitudes de cambio |
| Architecture-compliant solutions | Soluciones conformes |

---

## Fase H: Gestion de Cambios

### Inputs

| Input | Descripcion |
|-------|-------------|
| Architecture Contracts | Contratos vigentes |
| Compliance Assessments | Evaluaciones de Fase G |
| Change Requests | Solicitudes pendientes |
| Architecture Repository | Contenido actualizado |

### Steps

1. Establecer proceso de realizacion de valor
2. Desplegar herramientas de monitoreo
3. Gestionar riesgos
4. Proveer analisis para Architecture Board
5. Desarrollar change requirements para cumplir objetivos
6. Gestionar proceso de governance
7. Activar proceso para implementar cambios

### Outputs

| Output | Descripcion |
|--------|-------------|
| Architecture updates | Actualizaciones a la arquitectura |
| Changes to framework and principles | Cambios al framework |
| New Request for Architecture Work | Nueva solicitud (si aplica) |

## Matriz de Trazabilidad Inputs-Outputs

```
Fase     | Input Principal            | Output Principal
---------|----------------------------|---------------------------
Prelim   | TOGAF, Estrategia         | Framework Adaptado
A        | Request, Framework         | Vision, Statement of Work
B        | Vision, Statement          | Business Architecture
C        | Vision, Business Arch      | Data & App Architecture
D        | Vision, SI Architecture    | Technology Architecture
E        | All Architectures          | Migration Strategy
F        | Migration Strategy         | Roadmap, Migration Plan
G        | Migration Plan             | Contracts, Compliance
H        | Contracts, Compliance      | Updates, New Requests
```

## Resumen

Cada fase del ADM sigue el patron:
1. **Inputs**: Que necesito para empezar?
2. **Steps**: Que actividades ejecuto?
3. **Outputs**: Que produzco?
