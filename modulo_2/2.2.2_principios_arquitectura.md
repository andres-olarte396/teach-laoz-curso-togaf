# Principios de Arquitectura

**Tiempo estimado**: 25 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Definicion del Alcance Organizacional (2.2.1)

---

## Por que importa este subtema?

Los principios de arquitectura son las "leyes fundamentales" que guian todas las decisiones arquitectonicas. Sin principios claros, cada arquitecto toma decisiones basadas en su propio criterio, resultando en una arquitectura incoherente.

Los principios se definen en la **Fase Preliminar** y se refinan en la **Fase A**.

---

## Que es un Principio de Arquitectura

> "Una declaracion cualitativa de intencion que debe cumplirse por la arquitectura. Define las reglas generales y guias para el uso y despliegue de recursos y activos de TI."

**Caracteristicas**:
- General (no especifica una tecnologia)
- Duradero (valido por anos)
- Claro (no ambiguo)
- Accionable (puede usarse para tomar decisiones)

---

## Estructura de un Principio

TOGAF define un formato estandar de 5 elementos:

```
PRINCIPIO: [Nombre corto y memorable]
=====================================

DECLARACION:
  Frase clara que establece la directriz.

JUSTIFICACION:
  Por que este principio es importante.
  Que problemas previene o beneficios trae.

IMPLICACIONES:
  Consecuencias de adoptar este principio.
  Que se debe hacer para cumplirlo.
  Que limitaciones impone.

METRICAS (opcional):
  Como medir el cumplimiento del principio.
```

---

## Ejemplo Completo

```
PRINCIPIO: Comprar Antes que Construir
======================================

DECLARACION:
  Cuando exista una solucion comercial que satisfaga
  al menos el 80% de los requisitos, se preferira
  adquirirla antes que desarrollar una solucion custom.

JUSTIFICACION:
  El desarrollo custom tiene costos ocultos elevados:
  mantenimiento, evolucion, soporte, documentacion.
  Las soluciones comerciales distribuyen estos costos
  entre multiples clientes y ofrecen actualizaciones
  continuas. Nuestro core business no es desarrollar
  software, sino [actividad del negocio].

IMPLICACIONES:
  - Todo proyecto debe incluir evaluacion de soluciones
    comerciales antes de proponer desarrollo custom.
  - Se requiere justificacion formal para desarrollo
    custom aprobada por Architecture Board.
  - Equipos de desarrollo deben capacitarse en
    configuracion e integracion, no solo en desarrollo.
  - Presupuesto debe incluir licencias ademas de desarrollo.
  - Las personalizaciones deben minimizarse para
    facilitar actualizaciones del producto.

METRICAS:
  - % de proyectos que evaluan opciones comerciales
  - Razon desarrollo custom / solucion comercial
  - Costo de mantenimiento de soluciones custom vs comerciales
```

## Categorias de Principios

### Principios de Negocio

Guian la alineacion de TI con el negocio.

| Principio | Declaracion |
|-----------|-------------|
| Continuidad del negocio | Las operaciones deben continuar a pesar de fallas de sistemas |
| Cumplimiento regulatorio | Toda solucion debe cumplir con regulaciones aplicables |
| Privacidad de datos | Los datos personales se recolectan solo cuando es necesario |

### Principios de Datos

Guian la gestion de datos como activo.

| Principio | Declaracion |
|-----------|-------------|
| Datos como activo | Los datos son un activo corporativo con valor de negocio |
| Datos compartidos | Los datos se comparten a traves de funciones empresariales |
| Fuente unica de verdad | Cada dato tiene una fuente autoritativa |
| Calidad de datos | Los datos deben ser precisos, oportunos y completos |

### Principios de Aplicaciones

Guian el portafolio de aplicaciones.

| Principio | Declaracion |
|-----------|-------------|
| Independencia tecnologica | Las apps no deben depender de tecnologia propietaria |
| Reutilizacion | Capacidades comunes se implementan una vez y se reutilizan |
| Facilidad de uso | Las interfaces deben ser intuitivas y consistentes |
| Interoperabilidad | Las aplicaciones deben poder intercambiar datos |

### Principios de Tecnologia

Guian la infraestructura y plataformas.

| Principio | Declaracion |
|-----------|-------------|
| Estandares abiertos | Se prefieren estandares abiertos sobre propietarios |
| Cambios controlados | Los cambios siguen un proceso gestionado |
| Seguridad por diseno | La seguridad se incorpora desde el inicio |

## Proceso de Definicion

### Paso 1: Recopilar Inputs

| Fuente | Tipo de principio |
|--------|-------------------|
| Estrategia de negocio | Principios de negocio |
| Politicas corporativas | Compliance, seguridad |
| Lecciones aprendidas | Que salio mal y que evitar |
| Mejores practicas industria | Estandares del sector |

### Paso 2: Redactar Borradores

```
Reglas para redaccion:
- Usar lenguaje positivo ("hacer X" en lugar de "no hacer Y")
- Ser especifico pero no prescriptivo
- Evitar jerga tecnica en exceso
- Que un ejecutivo no tecnico pueda entenderlo
```

### Paso 3: Validar y Aprobar

| Validador | Que verifica |
|-----------|--------------|
| Stakeholders de negocio | Alineacion con estrategia |
| Arquitectos | Viabilidad tecnica |
| Legal/Compliance | Coherencia con regulaciones |
| Architecture Board | Aprobacion final |

### Paso 4: Publicar y Comunicar

- Publicar en repositorio accesible
- Capacitar a equipos
- Incluir en onboarding de nuevos empleados

## Usando Principios en el ADM

```
Fase A: Architecture Vision
  - Validar que la vision no viole principios

Fases B, C, D: Arquitecturas de Dominio
  - Cada decision arquitectonica debe alinearse con principios
  - Documentar que principios aplican a cada componente

Fase E: Oportunidades y Soluciones
  - Evaluar opciones contra principios
  - "Comprar vs Construir" guiado por principio #7

Fase G: Governance
  - Compliance reviews verifican adherencia a principios
  - Excepciones requieren aprobacion formal
```
