# Analisis de Impacto Cross-Domain

**Tiempo estimado**: 25 minutos
**Nivel**: Intermedio-Avanzado
**Prerrequisitos**: Matrices de Trazabilidad (5.2.1)

---

## Por que importa este subtema?

Un cambio en un dominio de arquitectura casi siempre afecta a otros dominios. Si cambia un proceso de negocio, probablemente cambian aplicaciones y datos. Si se migra una tecnologia, afecta aplicaciones que la usan.

El analisis de impacto cross-domain previene sorpresas costosas durante la implementacion.

---

## Que es Analisis de Impacto Cross-Domain

> "El proceso de evaluar como un cambio propuesto en un dominio de arquitectura (Negocio, Datos, Aplicaciones, Tecnologia) afecta a los otros dominios."

**Objetivo**: Identificar todos los efectos en cascada de un cambio antes de implementarlo.

## Proceso de Analisis

### Paso 1: Identificar el cambio

```markdown
## Cambio Propuesto

**ID**: CHG-2026-015
**Tipo**: Cambio de proceso de negocio
**Descripcion**: Agregar aprobacion automatica para pedidos < $1000
**Dominio origen**: Negocio
**Sponsor**: VP Ventas
```

### Paso 2: Analizar impacto en cada dominio

```markdown
## Analisis de Impacto Cross-Domain

### Impacto en NEGOCIO (Dominio Origen)
| Elemento | Impacto | Detalle |
|----------|---------|---------|
| Proceso Aprobacion Pedidos | Modificar | Agregar decision automatica |
| Rol Aprobador | Modificar | Solo para pedidos > $1000 |
| KPI Tiempo Aprobacion | Positivo | Reduccion esperada 80% |

### Impacto en DATOS
| Elemento | Impacto | Detalle |
|----------|---------|---------|
| Entidad Pedido | Modificar | Nuevo campo: tipo_aprobacion |
| Regla de Negocio | Nueva | "Si monto < 1000 entonces auto-aprobar" |
| Auditoria | Modificar | Registrar aprobaciones automaticas |

### Impacto en APLICACIONES
| Elemento | Impacto | Detalle |
|----------|---------|---------|
| Order Service | Modificar | Implementar logica de auto-aprobacion |
| Motor de Reglas | Modificar | Nueva regla de aprobacion |
| Workflow Engine | Modificar | Nuevo path automatico |
| Notificaciones | Modificar | No notificar aprobador para auto |

### Impacto en TECNOLOGIA
| Elemento | Impacto | Detalle |
|----------|---------|---------|
| Base de datos | Schema change | ALTER TABLE pedidos |
| Kafka | Nuevo evento | "pedido-auto-aprobado" |
| Monitoreo | Actualizar | Alertas para tasa de auto-aprobacion |
```

### Paso 3: Evaluar riesgos

```markdown
## Evaluacion de Riesgos

| Riesgo | Dominio | Probabilidad | Impacto | Mitigacion |
|--------|---------|--------------|---------|------------|
| Fraude por auto-aprobacion | Negocio | Media | Alto | Monitoreo de patrones |
| Inconsistencia de datos | Datos | Baja | Medio | Validacion en servicio |
| Degradacion de rendimiento | Tecnologia | Baja | Bajo | Pruebas de carga |
```

### Paso 4: Estimar esfuerzo

```markdown
## Estimacion de Esfuerzo por Dominio

| Dominio | Componentes | Esfuerzo | Equipo |
|---------|-------------|----------|--------|
| Negocio | Proceso, Rol | 2 dias | Analista |
| Datos | Schema, Regla | 1 dia | Data Architect |
| Aplicaciones | 4 servicios | 2 semanas | Dev Team |
| Tecnologia | DB, Kafka, Monitoreo | 3 dias | DevOps |
| **Total** | | **~3 semanas** | |
```

```
          +---+-----------+-----------+-----------+-----------+
          |App| Bajo      | Medio     | -         | Alto      |
          +---+-----------+-----------+-----------+-----------+
          |Tec| Bajo      | Bajo      | Alto      | -         |
          +---+-----------+-----------+-----------+-----------+

```

### Ejemplo con valores

| Si cambia... | Neg | Dat | App | Tec | Ejemplo |
|--------------|-----|-----|-----|-----|---------|
| **Proceso de negocio** | - | A | A | M | Nueva aprobacion |
| **Modelo de datos** | M | - | A | M | Nueva entidad |
| **Aplicacion** | B | M | - | A | Reescribir servicio |
| **Plataforma** | B | B | A | - | Migrar a cloud |

## Ejemplo Completo: Migracion de CRM

### Cambio

"Migrar de Siebel CRM a Salesforce"

### Analisis Cross-Domain

```markdown
## Impacto en NEGOCIO

| Area | Impacto | Accion |
|------|---------|--------|
| Proceso de ventas | Alto | Redisenar para aprovechar Salesforce |
| Proceso de servicio | Alto | Adaptar a casos de Salesforce |
| Reportes gerenciales | Medio | Recrear dashboards |
| Capacitacion | Alto | 500 usuarios a capacitar |
| Roles | Medio | Nuevos admins de Salesforce |

**Esfuerzo negocio**: 3 meses de change management

## Impacto en DATOS

| Area | Impacto | Accion |
|------|---------|--------|
| Modelo de cliente | Alto | Mapear a modelo Salesforce |
| Historial de interacciones | Alto | Migrar 5 anos de data |
| Calidad de datos | Alto | Limpiar antes de migrar |
| Integracion MDM | Medio | Ajustar flujos de master data |
| Reportes analiticos | Medio | Ajustar ETL desde nueva fuente |

**Esfuerzo datos**: 2 meses de migracion y limpieza

## Impacto en APLICACIONES

| Area | Impacto | Accion |
|------|---------|--------|
| Portal de ventas | Alto | Reintegrar con Salesforce APIs |
| Sistema de cotizaciones | Alto | Usar Salesforce CPQ |
| Integracion ERP | Alto | Nueva integracion SAP-Salesforce |
| App movil | Alto | Usar Salesforce Mobile |
| BI/DWH | Medio | Nuevo conector Salesforce |
| 15 integraciones legacy | Alto | Rehacer todas |

**Esfuerzo aplicaciones**: 6 meses de desarrollo

## Impacto en TECNOLOGIA

| Area | Impacto | Accion |
|------|---------|--------|
| Servidores Siebel | Alto | Decomisionar |
| Base de datos Oracle | Medio | Reducir licencias |
| Integracion middleware | Alto | Adaptar MuleSoft |
| Seguridad/SSO | Medio | Integrar Salesforce con Okta |
| Monitoreo | Bajo | Agregar Salesforce a Datadog |
| Red/Firewall | Bajo | Abrir puertos a Salesforce |

**Esfuerzo tecnologia**: 2 meses de infraestructura

## Resumen Ejecutivo

| Dominio | Impacto | Esfuerzo | Riesgo |
|---------|---------|----------|--------|
| Negocio | Alto | 3 meses | Adopcion de usuarios |
| Datos | Alto | 2 meses | Perdida de datos en migracion |
| Aplicaciones | Muy Alto | 6 meses | Integraciones fallidas |
| Tecnologia | Medio | 2 meses | Bajo |
| **Total** | **Muy Alto** | **8-10 meses** | **Alto** |
```

## Checklist de Analisis

```markdown
## Checklist Analisis Cross-Domain

### Negocio
- [ ] Procesos afectados identificados
- [ ] Roles impactados documentados
- [ ] Capacitacion planificada
- [ ] KPIs ajustados

### Datos
- [ ] Entidades afectadas mapeadas
- [ ] Plan de migracion de datos
- [ ] Calidad de datos evaluada
- [ ] Impacto en reportes analizado

### Aplicaciones
- [ ] Aplicaciones afectadas listadas
- [ ] Integraciones mapeadas
- [ ] APIs a modificar identificadas
- [ ] Plan de pruebas definido

### Tecnologia
- [ ] Infraestructura afectada identificada
- [ ] Cambios de seguridad evaluados
- [ ] Monitoreo actualizado
- [ ] Plan de rollback definido
```
