# Modelado de Capacidades de Negocio

**Tiempo estimado**: 35 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Modulo 2 completo (ADM)

---

## Por que importa este subtema?

Las capacidades de negocio son el puente entre la estrategia y la arquitectura. Mientras los procesos describen "como" opera el negocio, las capacidades describen "que" puede hacer el negocio. Esta distincion es critica porque:

- Las capacidades son **estables** (cambian poco en decadas)
- Los procesos son **volatiles** (cambian con frecuencia)
- La arquitectura debe soportar capacidades, no solo procesos actuales

---

## Que es una Capacidad de Negocio

> "Una habilidad particular que un negocio puede poseer o intercambiar para lograr un proposito o resultado especifico."

**Caracteristicas**:
- Describe **que** hace el negocio, no **como**
- Es **estable** en el tiempo
- Es **unica** (no duplicada)
- Es **medible** en terminos de madurez

**Ejemplos**:
| Capacidad | Descripcion | Estabilidad |
|-----------|-------------|-------------|
| Gestion de Clientes | Habilidad de adquirir, retener y desarrollar clientes | Decadas |
| Procesamiento de Pedidos | Habilidad de recibir, validar y cumplir pedidos | Decadas |
| Gestion de Inventario | Habilidad de controlar existencias de productos | Decadas |

---

## Capacidad vs Proceso vs Funcion

| Concepto | Que describe | Ejemplo | Estabilidad |
|----------|--------------|---------|-------------|
| **Capacidad** | Que puede hacer | "Gestion de Pagos" | Alta (anos) |
| **Proceso** | Como lo hace | "Workflow de autorizacion de tarjeta" | Media (meses) |
| **Funcion** | Quien lo hace | "Departamento de Tesoreria" | Variable |

```
CAPACIDAD: Gestion de Pagos
     |
     +-- PROCESO 1: Pago con tarjeta credito
     |        |
     |        +-- FUNCION: Equipo de Tesoreria
     |        +-- SISTEMA: Gateway de pagos
     |
     +-- PROCESO 2: Pago por transferencia
     |        |
     |        +-- FUNCION: Equipo de Cuentas por Cobrar
     |        +-- SISTEMA: Modulo bancario ERP
     |
     +-- PROCESO 3: Pago en efectivo (tienda)
              |
              +-- FUNCION: Cajeros
              +-- SISTEMA: POS
```

## Estructura de un Mapa de Capacidades

### Niveles jerarquicos

```
NIVEL 1 (L1): Capacidades estrategicas (6-10)
    |
    +-- NIVEL 2 (L2): Capacidades tacticas (30-50 total)
            |
            +-- NIVEL 3 (L3): Capacidades operativas (100-200 total)
```

### Ejemplo: Retail

```
L1: GESTION DE CLIENTES
    |
    +-- L2: Adquisicion de Clientes
    |       +-- L3: Generacion de Leads
    |       +-- L3: Calificacion de Prospectos
    |       +-- L3: Conversion de Ventas
    |
    +-- L2: Servicio al Cliente
    |       +-- L3: Gestion de Consultas
    |       +-- L3: Gestion de Reclamos
    |       +-- L3: Soporte Tecnico
    |
    +-- L2: Retencion de Clientes
            +-- L3: Programas de Lealtad
            +-- L3: Recuperacion de Clientes
            +-- L3: Analisis de Churn

L1: GESTION DE PRODUCTOS
    |
    +-- L2: Desarrollo de Productos
    +-- L2: Gestion de Catalogo
    +-- L2: Pricing
    ...
```

## Proceso de Modelado

### Paso 1: Identificar capacidades L1

Fuentes:
- Cadena de valor de Porter
- Modelo operativo del negocio
- Estrategia corporativa

```
CADENA DE VALOR -> CAPACIDADES L1

Actividades Primarias:
- Logistica de entrada -> Gestion de Abastecimiento
- Operaciones -> Gestion de Produccion
- Logistica de salida -> Gestion de Distribucion
- Marketing y Ventas -> Gestion Comercial
- Servicio -> Gestion de Servicio al Cliente

Actividades de Soporte:
- Infraestructura -> Gestion Corporativa
- RRHH -> Gestion de Talento
- Tecnologia -> Gestion de Tecnologia
- Compras -> Gestion de Proveedores
```

### Paso 2: Descomponer en L2 y L3

Para cada L1, preguntar:
- Que sub-capacidades se necesitan?
- Son mutuamente excluyentes?
- Cubren todo el alcance de L1?

### Paso 3: Validar con stakeholders

- Negocio valida completitud
- TI valida mapeo a sistemas
- Estrategia valida alineacion

## Heat Map de Capacidades

Visualizacion que combina capacidades con evaluacion:

```
                    IMPORTANCIA ESTRATEGICA
                    Baja      Media     Alta
                 +----------+----------+----------+
            Alta |  INVEST  | INVEST   | PRIORIDAD|
                 |          |          |   MAXIMA |
   BRECHA   +----+----------+----------+----------+
   (Gap)    Media| MONITOREAR| MEJORAR | PRIORIDAD|
                 |          |          |          |
            +----+----------+----------+----------+
            Baja | MANTENER | MANTENER | PROTEGER |
                 |          |          |          |
                 +----------+----------+----------+

Codigo de colores:
- ROJO: Capacidad critica con brecha grande
- AMARILLO: Requiere atencion
- VERDE: Adecuado
```

## Mapeo Capacidad-Sistema

Vincular capacidades con sistemas que las soportan:

| Capacidad | Sistemas Actuales | Cobertura | Gaps |
|-----------|-------------------|-----------|------|
| Gestion de Clientes | Salesforce CRM | 80% | Segmentacion avanzada |
| Procesamiento Pedidos | SAP SD, Portal Web | 70% | Omnicanalidad |
| Gestion Inventario | SAP MM | 90% | Tiempo real |
| Gestion Pagos | Gateway externo | 95% | - |

Este mapeo es fundamental para la Fase C (Arquitectura de Aplicaciones).

## Errores Comunes

| Error | Consecuencia | Prevencion |
|-------|--------------|------------|
| Confundir capacidad con proceso | Mapa inestable | Preguntar "que" no "como" |
| Demasiadas capacidades L1 | Pierde valor estrategico | Maximo 10-12 L1 |
| Capacidades duplicadas | Confusion, gaps | Validar MECE (mutuamente excluyentes, colectivamente exhaustivas) |
| Sin metricas de madurez | No se puede priorizar | Definir escala 1-5 |
| No validar con negocio | Arquitectura desconectada | Workshops con stakeholders |
