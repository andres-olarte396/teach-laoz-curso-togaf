# Business Scenarios

**Tiempo estimado**: 25 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Gap Analysis de Negocio (3.2.1)

---

## Por que importa este subtema?

Los Business Scenarios son una tecnica de TOGAF para **derivar requisitos de arquitectura desde problemas reales de negocio**. En lugar de empezar con tecnologia, empezamos con:

- Un problema concreto que el negocio enfrenta
- Los actores involucrados
- El resultado deseado

Esto asegura que la arquitectura resuelve problemas reales, no imaginarios.

---

## Que es un Business Scenario

> "Una tecnica para descubrir y documentar requisitos de negocio, describiendo un problema o situacion que afecta a la organizacion y sus sistemas."

**Componentes**:
1. Problema u oportunidad de negocio
2. Ambiente de negocio (contexto)
3. Objetivos y medidas de exito
4. Actores y sus roles
5. Proceso o flujo deseado
6. Requisitos derivados

---

## Ejemplo Completo

```markdown
# Business Scenario: Cotizacion Rapida para Cliente Corporativo

## 1. Problema/Oportunidad

Los clientes corporativos esperan respuestas de cotizacion en 24 horas,
pero nuestro proceso actual toma 5 dias. Esto resulta en:
- Perdida del 30% de oportunidades por demora
- Frustracion de clientes (NPS -15 puntos)
- Costo de oportunidad de $2M anuales

## 2. Ambiente de Negocio

### 2.1 Contexto Organizacional
- Division: Ventas B2B
- Unidades involucradas: Ventas, Pricing, Credito, Operaciones
- Geografia: Nacional

### 2.2 Situacion Actual
- Vendedor recibe solicitud por email
- Crea cotizacion en Excel
- Envia a Pricing para validar precios (email, 1-2 dias)
- Envia a Credito para validar cliente (email, 1-2 dias)
- Genera documento final y envia al cliente

### 2.3 Drivers
- Competencia responde en 24 horas
- Objetivo estrategico de "Experiencia Digital"
- Presion de margen requiere eficiencia operativa

## 3. Objetivos

### 3.1 Objetivo de Negocio
Reducir el tiempo de cotizacion de 5 dias a menos de 4 horas
para el 80% de las solicitudes.

### 3.2 Medidas de Exito
| Metrica | Actual | Objetivo |
|---------|--------|----------|
| Tiempo promedio | 5 dias | <4 horas |
| % cotizaciones en <24h | 5% | 90% |
| Tasa de conversion | 20% | 35% |
| Errores de pricing | 8% | <1% |

## 4. Actores Humanos

| Actor | Rol | Concerns |
|-------|-----|----------|
| Ejecutivo de Cuenta | Captura solicitud, envia cotizacion | Rapidez, precision |
| Analista de Pricing | Valida precios especiales | Margenes, politicas |
| Analista de Credito | Valida riesgo del cliente | Riesgo, limites |
| Gerente de Ventas | Aprueba excepciones | Control, visibilidad |
| Cliente Corporativo | Recibe y evalua cotizacion | Precio, tiempo |

## 5. Actores de Sistema

| Sistema | Rol | Capacidad Requerida |
|---------|-----|---------------------|
| CRM (Salesforce) | Captura oportunidad | Formulario de cotizacion |
| ERP (SAP) | Fuente de precios | API de consulta de precios |
| Sistema de Credito | Valida cliente | Evaluacion automatica |
| Portal Web | Entrega cotizacion | Acceso cliente |

## 6. Flujo del Escenario

### 6.1 Flujo Principal
1. Cliente solicita cotizacion (email/portal/telefono)
2. Ejecutivo captura solicitud en CRM con productos y cantidades
3. Sistema consulta precios automaticamente del ERP
4. Sistema evalua credito del cliente automaticamente
5. Si todo OK: Sistema genera cotizacion y notifica a Ejecutivo
6. Ejecutivo revisa y envia al cliente via portal
7. Cliente accede al portal y ve su cotizacion
8. Sistema registra evento y actualiza pipeline

### 6.2 Flujos Alternativos
- **Precio especial requerido**: Sistema detecta descuento >15%,
  enruta a Analista de Pricing para aprobacion (SLA: 2 horas)
- **Cliente nuevo**: Sistema no tiene historial crediticio,
  solicita documentacion y enruta a Credito (SLA: 4 horas)
- **Producto no estandar**: Sistema detecta producto custom,
  enruta a Operaciones para validar disponibilidad

### 6.3 Excepciones
- **ERP no disponible**: Sistema usa cache de precios (max 24h),
  marca cotizacion como "sujeta a confirmacion"
- **Cliente bloqueado**: Sistema rechaza automaticamente,
  notifica a Ejecutivo con motivo

## 7. Requisitos Derivados

### 7.1 Requisitos de Negocio
- REQ-BUS-001: El proceso debe completarse en <4 horas para casos estandar
- REQ-BUS-002: Descuentos >15% requieren aprobacion de Pricing
- REQ-BUS-003: Clientes sin historial requieren evaluacion crediticia manual

### 7.2 Requisitos de Datos
- REQ-DAT-001: Precios deben estar disponibles en tiempo real desde ERP
- REQ-DAT-002: Historial crediticio de cliente debe ser consultable
- REQ-DAT-003: Catalogo de productos debe incluir flag de "estandar/custom"

### 7.3 Requisitos de Aplicacion
- REQ-APP-001: CRM debe tener formulario de captura de cotizacion
- REQ-APP-002: Debe existir integracion CRM-ERP para precios
- REQ-APP-003: Debe existir motor de reglas para enrutamiento
- REQ-APP-004: Portal debe permitir acceso a clientes

### 7.4 Requisitos de Tecnologia
- REQ-TEC-001: APIs del ERP deben soportar 100 consultas/minuto
- REQ-TEC-002: Sistema debe tener disponibilidad 99.5%
- REQ-TEC-003: Portal debe ser accesible desde movil
```

## Proceso de Creacion

```
1. IDENTIFICAR ESCENARIOS
   - Workshops con stakeholders de negocio
   - Analisis de pain points
   - Oportunidades estrategicas
         |
         v
2. DESARROLLAR ESCENARIOS
   - Documentar cada componente
   - Validar con actores involucrados
   - Iterar hasta consenso
         |
         v
3. DERIVAR REQUISITOS
   - Extraer requisitos de cada escenario
   - Categorizar por dominio
   - Consolidar duplicados
         |
         v
4. VALIDAR
   - Confirmar que requisitos abordan el escenario
   - Verificar completitud
   - Priorizar
```
