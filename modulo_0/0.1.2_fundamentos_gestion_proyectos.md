# Fundamentos de Gestion de Proyectos

**Tiempo estimado**: 25 minutos
**Nivel**: Basico
**Prerrequisitos**: Conceptos de TI Empresarial (0.1.1)

---

## Por que importa este subtema?

La arquitectura empresarial se implementa a traves de proyectos. Entender como funcionan los proyectos permite al arquitecto comunicarse efectivamente con gerentes de proyecto y entender las restricciones de implementacion.

---

## Que es un Proyecto

> "Un esfuerzo temporal emprendido para crear un producto, servicio o resultado unico." - PMI

**Caracteristicas**:
- Temporal (inicio y fin definidos)
- Unico (no es operacion repetitiva)
- Entregables especificos
- Restricciones (alcance, tiempo, costo)

---

## Triple Restriccion

```
                    ALCANCE
                      /\
                     /  \
                    /    \
                   /  OK  \
                  /________\
                 /          \
            TIEMPO -------- COSTO

"Bueno, rapido, barato: escoge dos"
```

### Interdependencia

| Si cambias... | Impacto en... |
|---------------|---------------|
| Mas alcance | +Tiempo o +Costo |
| Menos tiempo | -Alcance o +Costo |
| Menos costo | -Alcance o +Tiempo |

## Ciclo de Vida del Proyecto

```
+----------+   +----------+   +----------+   +----------+   +----------+
| INICIO   |-->|PLANEACION|-->| EJECUCION|-->| CONTROL  |-->|  CIERRE  |
+----------+   +----------+   +----------+   +----------+   +----------+
     |              |              |              |              |
     v              v              v              v              v
  Charter       Plan de        Trabajo        Monitoreo      Lecciones
  Stakeholders  proyecto       real           Cambios        Entrega
  Objetivos     Cronograma     Entregables    Reportes       Cierre
```

### Fases Detalladas

```markdown
## Inicio
- Definir objetivos del proyecto
- Identificar stakeholders
- Crear Project Charter
- Obtener aprobacion

## Planeacion
- Definir alcance detallado (WBS)
- Crear cronograma
- Estimar recursos y costos
- Identificar riesgos
- Plan de comunicacion

## Ejecucion
- Ejecutar el trabajo planificado
- Gestionar equipo
- Comunicar avances
- Gestionar proveedores

## Control
- Monitorear progreso
- Gestionar cambios
- Reportar estado
- Tomar acciones correctivas

## Cierre
- Entregar producto final
- Obtener aceptacion
- Documentar lecciones
- Liberar recursos
```

## Metodologias de Proyecto

### Waterfall (Cascada)

```
Requisitos --> Diseno --> Desarrollo --> Pruebas --> Despliegue

Caracteristicas:
+ Planificacion clara upfront
+ Documentacion completa
+ Facil de entender
- Poca flexibilidad
- Feedback tardio
- Riesgo alto si requisitos cambian
```

### Agile (Agil)

```
Sprint 1    Sprint 2    Sprint 3    Sprint 4
[Plan-Do]   [Plan-Do]   [Plan-Do]   [Plan-Do]
    |           |           |           |
    v           v           v           v
Increment   Increment   Increment   Producto
    1           2           3          Final

Caracteristicas:
+ Flexibilidad
+ Feedback continuo
+ Entregas incrementales
- Menos predecible
- Requiere involucramiento
- Documentacion puede sufrir
```

### Comparativa

| Aspecto | Waterfall | Agile |
|---------|-----------|-------|
| Requisitos | Fijos al inicio | Evolucionan |
| Planificacion | Detallada upfront | Por iteracion |
| Entrega | Al final | Incremental |
| Cambios | Costosos | Bienvenidos |
| Cliente | Al inicio y final | Continuo |
| Documentacion | Extensa | Minima viable |

## Roles en Proyectos

```markdown
## Roles Tipicos

### Sponsor
- Provee financiamiento
- Toma decisiones ejecutivas
- Remueve impedimentos organizacionales

### Project Manager
- Gestiona el proyecto dia a dia
- Coordina equipo y recursos
- Reporta avances
- Gestiona riesgos

### Equipo de Proyecto
- Ejecuta el trabajo tecnico
- Reporta progreso al PM
- Colabora con otros miembros

### Stakeholders
- Interesados en el resultado
- Proveen requisitos
- Aceptan entregables
```

## Relacion Arquitectura - Proyectos

### El Arquitecto en Proyectos

```
+------------------+
| ARQUITECTURA     |
| EMPRESARIAL      |
+--------+---------+
         |
         | Estandares
         | Guias
         | Patrones
         v
+------------------+     +------------------+
| PROYECTO A       |     | PROYECTO B       |
| - PM             |     | - PM             |
| - Arquitecto Sol |     | - Arquitecto Sol |
| - Equipo         |     | - Equipo         |
+------------------+     +------------------+
         |                        |
         v                        v
+------------------+     +------------------+
| SISTEMA A        |     | SISTEMA B        |
| (Cumple EA)      |     | (Cumple EA)      |
+------------------+     +------------------+
```

### Interacciones

| Fase Proyecto | Rol del Arquitecto |
|---------------|-------------------|
| Inicio | Validar alineacion con EA |
| Planeacion | Definir arquitectura de solucion |
| Ejecucion | Resolver dudas tecnicas |
| Control | Review de compliance |
| Cierre | Actualizar repositorio EA |
