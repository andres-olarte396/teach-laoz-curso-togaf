# Modelado y Diagramacion Basica

**Tiempo estimado**: 30 minutos
**Nivel**: Basico
**Prerrequisitos**: Fundamentos de Gestion de Proyectos (0.1.2)

---

## Por que importa este subtema?

Los arquitectos comunican ideas complejas a traves de modelos y diagramas. Entender los fundamentos de modelado permite crear y leer artefactos arquitectonicos efectivamente.

---

## Que es un Modelo

> "Una representacion simplificada de la realidad que captura los aspectos mas importantes para un proposito especifico."

**Caracteristicas de buenos modelos**:
- Simples: Eliminan complejidad innecesaria
- Claros: Faciles de entender
- Precisos: Representan la realidad fielmente
- Utiles: Sirven para tomar decisiones

## Notaciones Comunes

### Cajas y Flechas (Basico)

```
+----------+                    +----------+
|          |                    |          |
| Elemento |  --- relacion -->  | Elemento |
|    A     |                    |    B     |
+----------+                    +----------+

Reglas basicas:
- Cajas = Entidades (sistemas, componentes, actores)
- Flechas = Relaciones (flujos, dependencias, comunicacion)
- Etiquetas = Nombres descriptivos
```

### Ejemplo: Sistema Simple

```
+----------+     HTTP      +----------+     SQL      +----------+
|  Usuario |  ---------->  |  Web     |  -------->  | Database |
|  Browser |  <----------  |  Server  |  <--------  |          |
+----------+     HTML      +----------+    Data     +----------+
```

## UML Basico

### Que es UML

> "Unified Modeling Language: Lenguaje estandar para visualizar, especificar, construir y documentar sistemas de software."

### Diagrama de Casos de Uso

```
+------------------------------------------+
|           Sistema de Ventas              |
|  +----------------------------------+    |
|  |                                  |    |
|  |   (Crear Pedido)                 |    |
|  |        |                         |    |
|  |   (Ver Catalogo)                 |    |
|  |        |                         |    |
|  |   (Procesar Pago)                |    |
|  |                                  |    |
|  +----------------------------------+    |
+------------------------------------------+
        |
    +-------+
    |Cliente|  (Actor)
    +-------+

Elementos:
( ) = Caso de uso (funcionalidad)
[ ] = Sistema (limite del sistema)
Stick figure = Actor (usuario o sistema externo)
```

### Diagrama de Componentes

```
+------------------+      +------------------+
|   <<component>>  |      |   <<component>>  |
|   Web Frontend   |----->|   API Backend    |
+------------------+      +--------+---------+
                                   |
                          +--------v---------+
                          |   <<component>>  |
                          |   Database       |
                          +------------------+

Elementos:
<<stereotype>> = Tipo de elemento
---> = Dependencia/uso
```

### Diagrama de Secuencia

```
Usuario      Frontend      Backend       Database
   |             |            |             |
   | click       |            |             |
   |------------>|            |             |
   |             | request    |             |
   |             |----------->|             |
   |             |            | query       |
   |             |            |------------>|
   |             |            |    data     |
   |             |            |<------------|
   |             |  response  |             |
   |             |<-----------|             |
   |   display   |            |             |
   |<------------|            |             |
   |             |            |             |

Elementos:
Lineas verticales = Linea de vida (actor/componente)
Flechas horizontales = Mensajes
Tiempo fluye de arriba hacia abajo
```

## BPMN Basico

### Que es BPMN

> "Business Process Model and Notation: Notacion grafica para modelar procesos de negocio."

### Elementos Basicos

```
EVENTOS                ACTIVIDADES           GATEWAYS

   O                   +--------+               <>
 Inicio               |  Tarea  |            Exclusivo
                      +--------+              (XOR)

  (O)                 [========]               <>
  Fin                 Sub-proceso           Paralelo
                                              (AND)

   O                                           <>
 Timer                                      Inclusivo
                                              (OR)
```

### Ejemplo: Proceso de Aprobacion

```
+----------------------------------------------------------+
|  Solicitar Compra                                         |
+----------------------------------------------------------+
|                                                           |
|    O -----> [Crear    ] -----> <> -----> [Aprobar ]      |
|  Inicio     Solicitud      Monto?        Gerente         |
|                              |                |           |
|                              | < $1000        |           |
|                              v                v           |
|                        [Auto-aprobar]    <> -----> (O)   |
|                              |         Aprobado?   Fin   |
|                              |              |             |
|                              +------<-------+             |
|                                     No                    |
|                                     |                     |
|                                     v                     |
|                               [Rechazar] ----> (O)       |
|                                                 Fin      |
+----------------------------------------------------------+

Simbolos:
O = Evento inicio
(O) = Evento fin
[ ] = Actividad
<> = Gateway (decision)
---> = Flujo de secuencia
```

## ArchiMate (Introduccion)

### Que es ArchiMate

> "Lenguaje de modelado para describir arquitectura empresarial, integrado con TOGAF."

### Capas de ArchiMate

```
+--------------------------------------------------+
|                 ESTRATEGIA                        |
|  Recursos, Capacidades, Cursos de Accion         |
+--------------------------------------------------+
                        |
+--------------------------------------------------+
|                   NEGOCIO                         |
|  Actores, Roles, Procesos, Servicios             |
+--------------------------------------------------+
                        |
+--------------------------------------------------+
|                 APLICACION                        |
|  Componentes, Servicios, Interfaces              |
+--------------------------------------------------+
                        |
+--------------------------------------------------+
|                 TECNOLOGIA                        |
|  Nodos, Dispositivos, Software de Sistema        |
+--------------------------------------------------+
```

### Elementos Basicos por Capa

```markdown
## Capa de Negocio
| Elemento | Simbolo | Uso |
|----------|---------|-----|
| Actor | Figura humana | Quien ejecuta |
| Rol | Figura con sombrero | Responsabilidad |
| Proceso | Flecha redondeada | Actividad |
| Servicio | Rectangulo redondeado | Lo que se ofrece |

## Capa de Aplicacion
| Elemento | Simbolo | Uso |
|----------|---------|-----|
| Componente | Rectangulo | Pieza de software |
| Servicio | Rectangulo redondeado | Funcionalidad expuesta |
| Interface | Linea con circulo | Punto de acceso |

## Capa de Tecnologia
| Elemento | Simbolo | Uso |
|----------|---------|-----|
| Nodo | Cubo 3D | Recurso computacional |
| Dispositivo | Cubo 3D con linea | Hardware |
| Software | Rectangulo | Software de sistema |
```

## Buenas Practicas de Diagramacion

### Principios

```markdown
## Reglas de Oro

### 1. Claridad sobre Completitud
- Mejor un diagrama simple que uno completo pero confuso
- Usar multiples diagramas si es necesario

### 2. Audiencia Apropiada
- Ejecutivos: Alto nivel, poco detalle
- Tecnicos: Mas detalle, precision

### 3. Consistencia
- Usar mismos simbolos para mismos conceptos
- Mantener estilo visual uniforme

### 4. Leyenda
- Siempre incluir leyenda si no es notacion estandar
- Explicar colores y simbolos especiales

### 5. Titulo y Contexto
- Titulo descriptivo
- Fecha de version
- Alcance del diagrama
```

### Errores Comunes

| Error | Problema | Solucion |
|-------|----------|----------|
| Demasiado detalle | Confuso, no se ve el bosque | Dividir en capas |
| Sin leyenda | No se entiende | Agregar leyenda |
| Flechas cruzadas | Dificil seguir | Reorganizar layout |
| Inconsistencia | Confunde | Usar plantillas |
| Sin contexto | No se sabe que es | Agregar titulo y scope |
