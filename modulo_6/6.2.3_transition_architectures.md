# Transition Architectures

**Tiempo estimado**: 25 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Architecture Roadmap (6.2.2)

---

## Por que importa este subtema?

El salto directo de Baseline a Target rara vez es posible o deseable. Las Transition Architectures son estados intermedios que:

- Permiten entregar valor incremental
- Reducen el riesgo de "big bang"
- Mantienen operaciones durante el cambio
- Permiten aprender y ajustar

---

## Que es una Transition Architecture

> "Una descripcion formal de la arquitectura empresarial en un estado intermedio entre el baseline y el target."

**Criterios de una buena transicion**:
- Operacionalmente viable (el negocio puede funcionar)
- Entrega valor propio (no solo "preparacion")
- Tiempo acotado (no permanece indefinidamente)
- Documentada igual que baseline/target

---

## Ejemplo: 3 Transiciones

```
BASELINE                                                      TARGET
+------------+                                            +------------+
| CRM Legacy |                                            | Salesforce |
| On-Premise |                                            | Cloud      |
| Manual     |                                            | Integrado  |
+------------+                                            +------------+
      |                                                         ^
      |   TRANSITION 1        TRANSITION 2        TRANSITION 3  |
      |   +------------+      +------------+      +------------+ |
      +-->| Legacy +   |----->| Salesforce |----->| Salesforce |-+
          | Salesforce |      | Primary    |      | 100%       |
          | (Piloto)   |      | Legacy RO  |      | Legacy OFF |
          +------------+      +------------+      +------------+

T1: Piloto con 20 usuarios, sync bidireccional
T2: Salesforce es sistema principal, Legacy solo lectura
T3: Legacy apagado, migracion completa
```

## Documentando una Transition Architecture

```markdown
# Transition Architecture 2
# CRM: Salesforce Primary

## Metadata
- Fecha efectiva: 2026-09-01
- Duracion estimada: 6 meses
- Precede a: Transition 3 / Target

## Estado de Componentes

### Aplicaciones
| Aplicacion | Estado | Notas |
|------------|--------|-------|
| Salesforce | ACTIVO (Primary) | Sistema de registro |
| Siebel | READ-ONLY | Solo consulta historica |
| Integracion Legacy | ACTIVO | Sync Salesforce -> Siebel |

### Datos
| Entidad | Source of Truth | Notas |
|---------|-----------------|-------|
| Cliente | Salesforce | Siebel recibe replica |
| Oportunidad | Salesforce | Siebel no tiene |
| Historico | Siebel | Solo lectura |

### Infraestructura
| Componente | Estado |
|------------|--------|
| Servers Siebel | Reducidos (2 de 8) |
| Salesforce Cloud | Produccion |
| Integracion MuleSoft | Activa |

## Valor Entregado
- 180 usuarios en Salesforce
- Vision 360 de cliente activo
- Reduccion 50% costo licencias Siebel

## Limitaciones Conocidas
- Historico > 2 anos solo en Siebel
- Reportes legacy aun necesitan Siebel
- Doble mantenimiento de integracion

## Criterios de Exit (para pasar a T3)
- [ ] 100% usuarios migrados
- [ ] Reportes historicos disponibles en SF
- [ ] Plan de decomision de Siebel aprobado

## Riesgos Especificos de Esta Transicion
| Riesgo | Mitigacion |
|--------|------------|
| Usuarios usan ambos sistemas | Capacitacion, restriccion de escritura en Siebel |
| Datos desincronizados | Monitoreo de integracion 24/7 |
```

## Planificacion de Transiciones

### Cuantas transiciones?

| Duracion del programa | Transiciones sugeridas |
|-----------------------|------------------------|
| < 6 meses | 0-1 (puede ir directo) |
| 6-12 meses | 1-2 |
| 12-24 meses | 2-3 |
| > 24 meses | 3-5 |

### Criterios para definir cortes

- Cambios de sistema principal
- Migraciones de datos significativas
- Cambios organizacionales
- Releases de funcionalidad clave
- Puntos de no retorno

```
   | New  |   | Old  |

   +------+   +------+

Gradualmente mas trafico va al nuevo sistema

```

## Integraciones Temporales

Las transiciones a menudo requieren integraciones que no existiran en el target:

```markdown
## Integraciones Temporales (T2)

| Integracion | Proposito | Fecha de retiro |
|-------------|-----------|-----------------|
| SF -> Siebel Sync | Mantener Siebel actualizado | T3 (Q1-2027) |
| Siebel Facade API | Reportes legacy | T3 (Q1-2027) |
| Dual-write Adapter | Consistencia temporal | T3 (Q1-2027) |

IMPORTANTE: Documentar fecha de retiro para evitar
que se vuelvan permanentes.
```

## Resumen

| Concepto | Descripcion |
|----------|-------------|
| Transition Architecture | Estado intermedio viable |
| Valor propio | Cada transicion entrega beneficio |
| Coexistencia | Sistemas viejos y nuevos conviven |
| Integraciones temporales | Puentes que se retiran |
| Criterios de exit | Condiciones para avanzar |
