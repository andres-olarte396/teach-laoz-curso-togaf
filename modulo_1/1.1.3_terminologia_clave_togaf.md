# Terminologia Clave TOGAF

**Tiempo estimado**: 25 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Los Cuatro Pilares de TOGAF (1.1.2)

---

## Por que importa este subtema?

TOGAF tiene un vocabulario tecnico preciso. Usar terminos incorrectamente genera confusion en equipos, documentos ambiguos y decisiones erroneas. Este glosario te permite comunicarte con precision en cualquier contexto de arquitectura empresarial.

La diferencia entre decir "architecture" vs "design", o "ABB" vs "SBB", puede cambiar completamente el significado de una conversacion tecnica.

---

## Terminos Fundamentales

### Architecture (Arquitectura)

**Definicion TOGAF**:
> "La estructura fundamental de un sistema, incorporada en sus componentes, sus relaciones entre si y con el entorno, y los principios que gobiernan su diseno y evolucion a lo largo del tiempo."

**Dos usos del termino**:

| Uso | Significado | Ejemplo |
|-----|-------------|---------|
| Descripcion | Documentacion de un sistema | "La arquitectura del sistema de pagos" |
| Disciplina | Practica profesional | "El equipo de arquitectura" |

---

### Enterprise (Empresa)

**Definicion TOGAF**:
> "La coleccion mas alta de organizaciones que tienen un conjunto comun de objetivos."

**Importante**: "Enterprise" no significa necesariamente "toda la compania". Puede ser:
- Una division o unidad de negocio
- Un consorcio de empresas
- Una cadena de valor completa (incluyendo proveedores y clientes)
- Un programa o proyecto grande

```
EJEMPLOS DE "ENTERPRISE" EN TOGAF
=================================

Empresa completa:
  "Microsoft como organizacion global"

Division:
  "Microsoft Azure (cloud business unit)"

Programa:
  "Iniciativa de Transformacion Digital 2025"

Ecosistema:
  "Red de concesionarios Toyota + proveedores Tier 1"
```

---

### Stakeholder (Interesado)

**Definicion TOGAF**:
> "Un individuo, equipo, organizacion, o clase de estos, que tiene un interes en un sistema."

**Categorias comunes**:

| Categoria | Ejemplos | Concerns tipicos |
|-----------|----------|------------------|
| Ejecutivos | CEO, CFO, CIO | ROI, riesgo, alineacion estrategica |
| Negocio | Gerentes de producto, Operaciones | Funcionalidad, usabilidad, eficiencia |
| Tecnicos | Desarrolladores, Ops, DBA | Implementabilidad, mantenibilidad |
| Externos | Reguladores, Clientes, Partners | Compliance, SLAs, integracion |

---

### Concern (Preocupacion/Interes)

**Definicion TOGAF**:
> "Un interes en un sistema relevante para uno o mas de sus stakeholders."

**Ejemplos de concerns**:

```
Stakeholder: CFO
Concerns:
  - "Cual es el costo total de propiedad?"
  - "Cual es el retorno de inversion?"
  - "Que riesgos financieros existen?"

Stakeholder: Security Officer
Concerns:
  - "Como se protegen los datos sensibles?"
  - "Cumplimos con GDPR/PCI-DSS?"
  - "Cual es el plan de recuperacion ante desastres?"

Stakeholder: Development Team
Concerns:
  - "Que APIs debo consumir?"
  - "Que stack tecnologico usar?"
  - "Como se despliega a produccion?"
```

### Capability (Capacidad)

**Definicion TOGAF**:
> "Una habilidad que una organizacion, persona, o sistema posee."

**Tipos**:

| Tipo | Descripcion | Ejemplo |
|------|-------------|---------|
| Business Capability | Lo que el negocio puede hacer | "Gestionar relaciones con clientes" |
| Architecture Capability | Capacidad de hacer arquitectura | "Equipo de arquitectos certificados" |

**Importante**: Las capacidades son **abstractas y estables**. Los procesos y sistemas que las implementan pueden cambiar, pero la capacidad permanece.

```
CAPACIDAD vs PROCESO vs SISTEMA

Capacidad: "Procesamiento de pagos"
           (estable por decadas)
                |
                v
Proceso:   "Workflow de autorizacion de tarjeta"
           (cambia cada pocos anos)
                |
                v
Sistema:   "Aplicacion de gateway de pagos v3.2"
           (cambia frecuentemente)
```

---

### Building Block

**Definicion TOGAF**:
> "Un componente potencialmente reutilizable de negocio, TI, o capacidad arquitectonica que puede ser combinado con otros building blocks para entregar arquitecturas y soluciones."

**Dos tipos**:

| Tipo | Sigla | Nivel | Ejemplo |
|------|-------|-------|---------|
| Architecture Building Block | ABB | Conceptual | "Servicio de autenticacion" |
| Solution Building Block | SBB | Implementacion | "Okta Identity Cloud" |

```
ABB --> SBB
"Servicio de mensajeria asincrona" --> "Apache Kafka 3.0"
"Base de datos relacional" --> "PostgreSQL 15"
"Orquestador de contenedores" --> "Kubernetes 1.28"
```

---

### Deliverable, Artifact, Building Block

Tres conceptos que frecuentemente se confunden:

| Concepto | Que es | Ejemplo |
|----------|--------|---------|
| **Deliverable** | Producto de trabajo contractual que se revisa y aprueba formalmente | Architecture Definition Document |
| **Artifact** | Producto de trabajo mas granular que compone un deliverable | Diagrama de contexto, Catalogo de aplicaciones |
| **Building Block** | Componente reutilizable de la arquitectura misma | Servicio de logging, Componente de UI |

```
RELACION ENTRE CONCEPTOS
========================

Deliverable: "Architecture Definition Document"
|
+-- Contiene Artifacts:
    |-- Catalogo de Aplicaciones
    |-- Matriz Aplicacion-Datos
    |-- Diagrama de Integracion
    |
    +-- Los artifacts describen Building Blocks:
        |-- ABB: "Servicio de Clientes"
        |-- ABB: "Motor de Reglas de Negocio"
        |-- ABB: "Data Lake Analitico"
```

### Baseline vs Target Architecture

| Termino | Definicion | Pregunta que responde |
|---------|------------|----------------------|
| **Baseline** | Arquitectura existente (estado actual) | "Donde estamos hoy?" |
| **Target** | Arquitectura objetivo (estado futuro) | "Donde queremos estar?" |
| **Transition** | Arquitectura intermedia | "Como llegamos ahi?" |

```
Timeline:
=========

Hoy            +6 meses        +12 meses       +18 meses
 |                |                |                |
 v                v                v                v
BASELINE      TRANSITION 1    TRANSITION 2      TARGET
(Legacy)      (Hibrido)       (Cloud-First)   (Cloud-Native)
```

---

### Gap

**Definicion TOGAF**:
> "Una declaracion de diferencia entre dos estados."

**Formula conceptual**:
```
Gap = Target - Baseline

Tipos de gaps:
- Nuevo: Existe en Target, no en Baseline
- Eliminado: Existe en Baseline, no en Target
- Modificado: Existe en ambos pero es diferente
```

### View y Viewpoint

| Termino | Definicion | Analogia |
|---------|------------|----------|
| **Viewpoint** | Especificacion de como construir una vista | Plantilla/Formato |
| **View** | Representacion de arquitectura desde una perspectiva | Documento concreto |

```
Viewpoint: "Technology Portfolio Viewpoint"
  - Stakeholders: CIO, Enterprise Architect
  - Concerns: Stack tecnologico, obsolescencia, costos
  - Model kinds: Matriz de tecnologias, Diagrama de plataforma
           |
           | se instancia como
           v
View: "Technology Portfolio View - Q1 2026"
  - Muestra tecnologias actuales de Empresa XYZ
  - Incluye matriz de 47 tecnologias con estado de cada una
  - Diagrama de plataforma AWS + on-premise
```

### Principle (Principio)

**Definicion TOGAF**:
> "Una declaracion de direccion fundamental general destinada a guiar la toma de decisiones."

**Estructura de un principio**:

```
PRINCIPIO: Reutilizacion sobre Reinvencion
==========================================

Nombre: Reutilizacion sobre Reinvencion

Declaracion:
  Antes de construir cualquier componente nuevo, se debe
  evaluar si existe un componente existente que pueda
  reutilizarse o adaptarse.

Justificacion:
  La reutilizacion reduce costos, acelera entregas,
  y mejora la consistencia entre sistemas.

Implicaciones:
  - Se requiere un catalogo actualizado de componentes
  - Los equipos deben consultar el catalogo antes de disenar
  - Excepciones requieren aprobacion del Architecture Board
```

## Resumen

La terminologia TOGAF es precisa y no intercambiable. Los errores mas comunes:

| Error comun | Correccion |
|-------------|------------|
| Usar "arquitectura" y "diseno" indistintamente | Arquitectura = decisiones de alto impacto; Diseno = detalles de implementacion |
| Confundir ABB con SBB | ABB = conceptual ("servicio de auth"); SBB = producto ("Okta") |
| Tratar Baseline como ideal | Baseline = realidad actual, incluyendo deuda tecnica |
| Omitir viewpoints al crear vistas | Toda view debe basarse en un viewpoint definido |
