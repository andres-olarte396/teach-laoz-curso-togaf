# Baseline vs Target Architecture

**Tiempo estimado**: 25 minutos
**Nivel**: Intermedio
**Prerrequisitos**: Arquitectura vs Diseno (1.2.1)

---

## Por que importa este subtema?

Todo proyecto de arquitectura es un viaje de A a B. Si no sabes donde estas (Baseline) o a donde vas (Target), no puedes planificar el camino. Muchos proyectos fracasan porque:

1. **Baseline idealizado**: Documentan como "deberia ser" en lugar de como "es realmente"
2. **Target sin ancla**: Definen un futuro desconectado de la realidad actual
3. **Sin transiciones**: Saltan de hoy a 3 anos sin estados intermedios

---

## Definiciones

### Baseline Architecture (Estado Actual)

> "Un conjunto de especificaciones de la arquitectura existente. Es el punto de partida contra el cual se mide el progreso."

**Caracteristicas del baseline**:
- Documenta **lo que existe**, no lo que deberia existir
- Incluye sistemas legacy, deuda tecnica, workarounds
- Puede ser imperfecto, incompleto, o incoherente
- Es verificable contra la realidad operativa

### Target Architecture (Estado Objetivo)

> "Un conjunto de especificaciones que describen el estado futuro deseado de la arquitectura."

**Caracteristicas del target**:
- Representa la **vision** de lo que se quiere lograr
- Debe estar alineado con la estrategia de negocio
- Es aspiracional pero alcanzable
- Define exito medible

### Transition Architecture (Estado Intermedio)

> "Una arquitectura que describe un estado intermedio entre la baseline y la target."

**Caracteristicas**:
- Permite cambio incremental en lugar de "big bang"
- Cada transicion debe ser operacionalmente viable
- Puede haber multiples transiciones
- Cada una tiene valor propio (no solo "paso hacia el target")

## Documentando el Baseline

### Que incluir

```
BASELINE ARCHITECTURE - SISTEMA DE PEDIDOS
==========================================

1. INVENTARIO DE APLICACIONES
   - APP-001: Sistema de Pedidos (Java 8, Tomcat 8.5)
     Estado: Operativo, sin soporte de vendor
     Usuarios: 500
     Integraciones: 12

   - APP-002: Modulo de Facturacion (COBOL, Mainframe)
     Estado: Legacy, conocimiento limitado
     Usuarios: 50
     Integraciones: 3

2. INFRAESTRUCTURA ACTUAL
   - 2 servidores fisicos on-premise (HP ProLiant DL380 Gen9)
   - Base de datos Oracle 11g (EOL)
   - Sin redundancia geografica

3. INTEGRACIONES
   - APP-001 <-> APP-002: Archivos planos nocturnos
   - APP-001 <-> ERP: Web service SOAP
   - APP-002 <-> Banco: FTP con PGP

4. PROBLEMAS CONOCIDOS
   - 30% de pedidos requieren intervencion manual
   - Tiempo de respuesta promedio: 4.5 segundos
   - 3 caidas no planificadas por mes
   - Documentacion desactualizada (ultima revision 2019)

5. DEUDA TECNICA
   - Java 8 sin path de upgrade definido
   - Oracle 11g fuera de soporte
   - 40% del codigo sin tests automatizados
   - Credenciales hardcoded en configuracion
```

### Errores comunes al documentar baseline

| Error | Problema | Solucion |
|-------|----------|----------|
| Baseline idealizado | "CRM con datos limpios" cuando hay 40% duplicados | Documentar realidad, incluida la fea |
| Baseline incompleto | Omitir sistemas shadow IT | Incluir TODO lo que existe |
| Baseline estatico | Documentar una vez y olvidar | Actualizar periodicamente |
| Sin metricas | "Funciona bien" sin datos | Incluir metricas de rendimiento, disponibilidad |

## Documentando el Target

### Que incluir

```
TARGET ARCHITECTURE - SISTEMA DE PEDIDOS (18 MESES)
===================================================

1. VISION
   "Sistema de pedidos cloud-native con procesamiento
    en tiempo real, 99.9% disponibilidad, y experiencia
    omnicanal unificada."

2. APLICACIONES TARGET
   - Microservicio de Pedidos (Kotlin, Kubernetes)
     Objetivo: Procesamiento <500ms, auto-scaling

   - Microservicio de Facturacion (migrado de COBOL)
     Objetivo: Eliminar dependencia mainframe

3. INFRAESTRUCTURA TARGET
   - AWS EKS (multi-AZ)
   - RDS PostgreSQL (Multi-AZ, auto-failover)
   - Redundancia geografica activa-activa

4. INTEGRACIONES TARGET
   - Event-driven via Apache Kafka
   - APIs REST estandarizadas (OpenAPI 3.0)
   - Sin transferencias batch (todo tiempo real)

5. METRICAS OBJETIVO
   - 0% intervencion manual en flujo normal
   - Tiempo de respuesta p95 < 500ms
   - 99.9% disponibilidad (8.7h downtime/ano max)
   - Cobertura de tests > 80%

6. ALINEACION ESTRATEGICA
   - OKR-2026-01: Mejorar experiencia de cliente
   - OKR-2026-03: Reducir costos operativos 20%
```

### Criterios de un buen Target

| Criterio | Pregunta | Ejemplo |
|----------|----------|---------|
| Especifico | Que exactamente? | "Microservicios en K8s", no "modernizar" |
| Medible | Como sabemos si lo logramos? | "p95 < 500ms", no "mas rapido" |
| Alcanzable | Es realista? | Considerar capacidad y presupuesto |
| Relevante | Apoya la estrategia? | Vinculado a OKRs/objetivos |
| Temporal | Para cuando? | Fecha especifica |

## Gap Analysis

### Formula

```
GAP = TARGET - BASELINE

Para cada componente:
  - NUEVO: Existe en Target, no en Baseline
  - RETIRAR: Existe en Baseline, no en Target
  - MODIFICAR: Existe en ambos pero diferente
  - MANTENER: Existe en ambos, sin cambios
```

### Ejemplo de Gap Analysis

| Componente | Baseline | Target | Gap | Accion |
|------------|----------|--------|-----|--------|
| CRM | Siebel on-prem | Salesforce Cloud | NUEVO | Migrar datos, reentrenar usuarios |
| Facturacion | COBOL Mainframe | Microservicio Kotlin | MODIFICAR | Reescribir, mantener logica |
| ERP | SAP ECC | SAP ECC | MANTENER | Solo actualizar integraciones |
| Reportes Ad-hoc | Excel manual | - | RETIRAR | Reemplazar con BI self-service |
| BI Platform | - | PowerBI | NUEVO | Implementar, integrar |

## Transition Architectures

### Cuando son necesarias

- Proyectos de mas de 6 meses
- Cuando el target requiere cambios en multiple sistemas
- Cuando se necesita valor incremental
- Cuando el riesgo de "big bang" es inaceptable

### Disenando transiciones

```
PRINCIPIOS PARA TRANSICIONES
============================

1. Cada transicion debe ser operacionalmente viable
   - No dejar el sistema en estado inconsistente
   - Rollback posible si algo falla

2. Cada transicion debe entregar valor
   - No solo "preparacion" sin beneficio
   - Metricas mejoran en cada transicion

3. Minimizar integraciones temporales
   - Evitar "puentes" que luego hay que eliminar
   - Si se necesitan, documentar fecha de retiro

4. Considerar capacidad de cambio
   - No sobrecargar a los equipos
   - Balancear con operaciones normales
```

### Ejemplo: 3 Transiciones para migracion CRM

```
BASELINE (Hoy)
+------------------+
| Siebel CRM       |
| - 100K clientes  |
| - 200 usuarios   |
+------------------+

TRANSICION 1 (+4 meses): Piloto
+------------------+     +------------------+
| Siebel CRM       | <-> | Salesforce       |
| - 100K clientes  |     | - Sync bi-dir    |
| - 180 usuarios   |     | - 20 usuarios    |
+------------------+     +------------------+
Valor: Validar Salesforce con grupo piloto

TRANSICION 2 (+8 meses): Expansion
+------------------+     +------------------+
| Siebel CRM       | <-> | Salesforce       |
| - Solo historico |     | - 100K clientes  |
| - 20 usuarios    |     | - 180 usuarios   |
+------------------+     +------------------+
Valor: 90% de usuarios migrados

TARGET (+12 meses): Completo
                         +------------------+
                         | Salesforce       |
                         | - 100K clientes  |
                         | - 200 usuarios   |
                         | - Siebel retirado|
                         +------------------+
Valor: Sistema unificado, Siebel decomisionado
```

## Resumen

| Concepto | Proposito | Pregunta clave |
|----------|-----------|----------------|
| **Baseline** | Punto de partida | Donde estamos hoy, realmente? |
| **Target** | Destino | Donde queremos estar? |
| **Gap** | Diferencia | Que falta para llegar? |
| **Transition** | Camino | Como llegamos de forma incremental? |
